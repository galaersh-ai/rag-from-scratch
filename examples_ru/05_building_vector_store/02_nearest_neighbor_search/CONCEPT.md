# Алгоритмы поиска ближайших соседей — Концептуальный обзор

## Что такое поиск ближайших соседей?

**Поиск ближайшего соседа (NN)** — это задача нахождения наиболее похожих элементов на заданный запрос в коллекции точек данных.

В контексте векторных баз данных это означает поиск векторов, которые являются **наиболее близкими** к вектору запроса в многомерном пространстве.

### Основная задача

**Дано:**
- Коллекция из N векторов (документов)
- Вектор запроса
- Число k

**Найти:** k векторов, наиболее похожих на вектор запроса.

**Пример:**
```javascript
// You have 10,000 document embeddings
// User searches: "machine learning tutorials"
// Goal: Find the 5 most relevant documents
const results = await nearestNeighborSearch(queryVector, k=5);
```

---

## Почему поиск ближайших соседей важен

### Основа векторного поиска

Каждая операция векторной базы данных основывается на поиске ближайших соседей:

1. **Семантический поиск**: «Найти документы, похожие на этот запрос»
2. **Рекомендации**: «Найти товары, похожие на те, которые понравились пользователю»
3. **Кластеризация**: «Сгруппировать похожие элементы»
4. **Обнаружение аномалий**: «Найти элементы, которые не вписываются в паттерн»

### Практическое значение

**Без эффективного поиска ближайших соседей:**
- Сравнение 1 запроса с 10 000 документов: ~500 мс
- 100 запросов в секунду: нужно 500 серверов!

**С эффективным поиском ближайших соседей (HNSW):**
- Сравнение 1 запроса с 10 000 документов: ~50 мс
- 100 запросов в секунду: нужно 5 серверов!

**Ускорение в 10 раз = экономия затрат в 10 раз!**

## Задача k ближайших соседей (kNN)

### Понимание k

**k** — это количество соседей, которое вы хотите найти.

```
k=1:  Найти один наиболее похожий элемент
k=5:  Найти 5 наиболее похожих элементов
k=10: Найти 10 наиболее похожих элементов
```

### Визуализация kNN

Представьте двумерный график, где каждая точка — это документ:

```
            Query (Q)
               *
              / \
             /   \
        k=1 /     \ k=2
           /       \
          /         \
      Doc A         Doc B
        •             •
                         
                         
    Doc C        Doc D
      •            •
```

**k=1**: Возвращает только Doc A (ближайший)
**k=2**: Возвращает Doc A и Doc B
**k=3**: Возвращает Doc A, Doc B и либо Doc C, либо Doc D

### Почему k важен

**Слишком мало (k=1-2):**
- ❌ Можно пропустить релевантные результаты
- ❌ Нет разнообразия
- ✅ Очень сфокусированный поиск
- ✅ Максимально быстро

**Оптимально (k=3-10):**
- ✅ Хороший баланс
- ✅ Некоторое разнообразие
- ✅ Возможность фильтрации
- ✅ Достаточно быстро

**Слишком много (k=50+):**
- ❌ Много нерелевантных результатов
- ❌ Больше шума для фильтрации
- ✅ Максимальное покрытие
- ⚠️ Немного медленнее

## Точный и приближённый поиск

### Компромисс

**Точный поиск (полный перебор):**
- Сравнивает запрос с **каждым** вектором
- Гарантирует нахождение настоящих ближайших соседей
- Медленно: сложность O(N)

**Приближённый поиск (HNSW):**
- Использует изящные алгоритмы для пропуска сравнений
- Находит приближённых ближайших соседей (>95% точность)
- Быстро: сложность O(log N)

### Сравнение масштабируемости

```
Документы   | Точный поиск | Приближённый | Ускорение
------------|--------------|-------------|--------
10          | 1 мс         | 1 мс        | 1x
100         | 10 мс        | 2 мс        | 5x
1 000       | 100 мс       | 5 мс        | 20x
10 000      | 1 000 мс     | 10 мс       | 100x
100 000     | 10 000 мс    | 20 мс       | 500x
1 000 000   | 100 000 мс   | 30 мс       | 3 333x
```

**Ключевой вывод:** Разрыв резко возрастает с увеличением объёма данных!

### Когда использовать каждый метод

**Точный поиск:**
- ✅ Небольшие наборы данных (< 1 000 документов)
- ✅ Требуется 100% точность
- ✅ Валидация и тестирование
- ✅ Юридические/регуляторные требования

**Приближённый поиск:**
- ✅ Большие наборы данных (> 1 000 документов)
- ✅ Требования к времени отклика в реальном времени
- ✅ Продакшн-системы
- ✅ Точность 95–99% достаточна

**Гибридный подход:**
- Используйте приближённый поиск для кандидатов (быстро, k=50)
- Используйте точный поиск для переранжирования топ-результатов (точно, k=5)
- Лучшее из обоих миров!

## Алгоритм HNSW

### Что такое HNSW?

**Иерархическая обозримая малая мир (HNSW)** — это графовый алгоритм для приближённого поиска ближайших соседей.

Это самый популярный алгоритм, используемый в современных векторных базах данных.

### Ключевые концепции

**1. Графовая структура**

Вместо сравнения с каждым вектором HNSW строит граф, в котором:
- Каждый вектор — это узел
- Рёбра соединяют похожие векторы
- Несколько уровней создают иерархию

```
Layer 2:    A -------- B          (Few nodes, long edges)
             \        /
              \      /
Layer 1:   A --- C --- B --- D    (More nodes, medium edges)
            |   |   |   |
Layer 0: A-X-C-Y-B-Z-D-W-E-...   (All nodes, short edges)
```

**2. Обозримый малый мир**

«Малый мир» означает, что до любого узла можно добраться за несколько шагов:
- Начните с точки входа
- Жадно перемещайтесь к ближайшему соседу
- Повторяйте, пока не сможете стать ближе
- Быстрая навигация: O(log N)

**3. Иерархические уровни**

Несколько уровней позволяют выполнять поиск от грубого к точному:
1. Начните с верхнего уровня (разреженный, быстрая навигация)
2. Быстро приблизьтесь к цели
3. Перейдите на нижний уровень (более плотный, более точный)
4. Уточните поиск на нижнем уровне
5. Верните k ближайших соседей

### Как работает поиск в HNSW

**Пошагово:**

```
Query: "machine learning"
k = 3

1. Start at top layer entry point
   Current: Doc_A (similarity: 0.3)

2. Check neighbors: Doc_B (0.5), Doc_C (0.4)
   Move to Doc_B (better)

3. No better neighbors at this layer
   Drop to next layer

4. Check more neighbors: Doc_D (0.6), Doc_E (0.7)
   Move to Doc_E (better)

5. Drop to bottom layer

6. Check all neighbors: Doc_F (0.8), Doc_G (0.75), Doc_H (0.7)
   
7. Return top 3: Doc_F (0.8), Doc_G (0.75), Doc_E (0.7)
```

**Результат:** Найдены очень хорошие совпадения с помощью всего ~10 сравнений вместо 10 000!

### Характеристики производительности HNSW

**Время построения:**
- O(N log N) для построения индекса
- Медленнее, чем полный перебор, при вставке
- Разовые затраты

**Время поиска:**
- O(log N) на запрос
- Значительно быстрее полного перебора
- Хорошо масштабируется с объёмом данных

**Память:**
- Выше, чем у полного перебора (графовая структура)
- ~30% накладных расходов на рёбра
- По-прежнему помещается в оперативную память для миллионов векторов

**Точность:**
- Обычно 95–99% полноты
- Настраивается (можно обменять скорость на точность)
- Достаточно хорошо для большинства приложений

## Метрики расстояний

### Что такое метрика расстояний?

**Метрика расстояний** (или метрика сходства) измеряет, насколько похожи два вектора.

Различные метрики имеют различные свойства и области применения.

### Косинусное сходство

**Формула:**
```
similarity = (A · B) / (||A|| × ||B||)
```

**Что измеряет:** Угол между двумя векторами, игнорируя величину.

**Диапазон:** от -1 до 1
- 1.0 = Идентичное направление (максимальное сходство)
- 0.0 = Ортогональные (не связаны)
- -1.0 = Противоположные направления (не похожи)

**Наглядная интуиция:**
```
Vector A: →
Vector B: ↗     (Small angle = high similarity)
Vector C: ↑     (Medium angle = medium similarity)
Vector D: ←     (Large angle = low similarity)
```

**Почему используется:**
- **Направление важнее длины** для эмбеддингов
- Нормализуется по величине вектора
- Стандарт в NLP и семантическом поиске
- Хорошо работает с эмбеддингами

**Пример:**
```javascript
embed("dog")    · embed("puppy")   = 0.85  // High similarity
embed("dog")    · embed("cat")     = 0.60  // Medium similarity
embed("dog")    · embed("car")     = 0.15  // Low similarity
```

### Евклидово расстояние

**Формула:**
```
distance = √(Σ(A[i] - B[i])²)
```

**Что измеряет:** Расстояние по прямой между двумя точками.

**Диапазон:** от 0 до ∞
- 0.0 = Идентичные векторы
- Большие значения = Более различные

**Наглядная интуиция:**
```
A •----------• B    (Large distance)
A •-• C            (Small distance)
```

**Когда использовать:**
- Когда величина имеет значение
- Эмбеддинги изображений
- Координатные данные
- Научные вычисления

**Сравнение с косинусным:**
```
Vectors: A = [3, 4], B = [6, 8], C = [1, 0]

Cosine similarity:
  A vs B: 1.0  (Same direction, different magnitude)
  A vs C: 0.6  (Different direction)

Euclidean distance:
  A vs B: 5.0  (Far apart in space)
  A vs C: 4.1  (Closer in space)
```

### Какую метрику выбрать?

**Используйте косинусное сходство, когда:**
- Работаете с текстовыми эмбеддингами
- Направление важнее величины
- Векторы нормализованы
- Цель — семантическое сходство

**Используйте евклидово расстояние, когда:**
- Величина имеет значение
- Работаете с координатами
- Векторы изображений
- Научные данные

**Значение по умолчанию:** Косинусное сходство для семантического поиска (то, что использует embedded-vector-db).

## Оптимизация производительности

### Узкое место: Генерация эмбеддингов

**Разбивка времени для типичного поиска:**

```
Total time: 50ms
├── Embedding generation: 45ms (90%)
└── kNN search: 5ms (10%)
```

**Ключевой вывод:** Оптимизация поиска с 5 мс до 1 мс экономит лишь 4 мс, но кэширование эмбеддингов экономит 45 мс!

### Стратегии оптимизации

#### 1. Кэширование эмбеддингов

**Проблема:** Повторная генерация эмбеддингов для одного и того же запроса тратит время.

**Решение:**
```javascript
const cache = new Map();

async function search(query, k) {
    if (!cache.has(query)) {
        cache.set(query, await embed(query));
    }
    return await knnSearch(cache.get(query), k);
}
```

**Ускорение:**
- Первый поиск: 50 мс (45 мс эмбеддинг + 5 мс поиск)
- Поиски из кэша: 5 мс (0 мс эмбеддинг + 5 мс поиск)
- **В 10 раз быстрее!**

#### 2. Пакетная обработка

**Проблема:** Последовательная обработка медленная.

**Последовательно:**
```javascript
for (const query of queries) {
    await search(query);
}
// Time: N × 50ms
```

**Параллельно:**
```javascript
await Promise.all(queries.map(q => search(q)));
// Time: ~50ms (concurrent)
```

**Ускорение:**
- 4 запроса последовательно: 200 мс
- 4 запроса параллельно: 50 мс
- **В 4 раза быстрее!**

#### 3. Предварительное вычисление

**Проблема:** Генерация эмбеддингов во время поиска медленная.

**Решение:**
```javascript
// Offline: Pre-compute document embeddings
const docEmbeddings = await embedAllDocuments();
await saveToVectorDB(docEmbeddings);

// Online: Only embed query (1 vs N embeddings)
const queryEmbedding = await embed(query);
const results = await search(queryEmbedding);
```

**Почему это работает:**
- Документы эмбеддингируются один раз (офлайн)
- Только запрос эмбеддингируется во время поиска
- Значительное ускорение для повторных запросов

#### 4. Избыточная выборка для фильтрации

**Проблема:** После фильтрации нужно k результатов.

**Наивный подход:**
```javascript
const results = await search(query, 5);
const filtered = results.filter(r => r.category === "AI");
// Might get 0-5 results
```

**Лучший подход:**
```javascript
const candidates = await search(query, 20);
const filtered = candidates
    .filter(r => r.category === "AI")
    .slice(0, 5);
// Always get up to 5 results
```

**Компромисс:**
- Немного больше времени на поиск (20 мс вместо 18 мс)
- Гарантированное получение достаточного количества отфильтрованных результатов
- Стоит того, когда фильтрация является частой операцией

## Качество поиска vs скорость

### Компромисс полноты и точности

**Точность:** Какая доля возвращённых результатов является релевантной?
```
Precision = Relevant Results / Total Results
```

**Полнота:** Какую долю релевантных документов мы нашли?
```
Recall = Found Relevant / Total Relevant
```

**Пример:**
```
10 relevant documents exist in database
Search returns 5 results: 4 relevant, 1 irrelevant

Precision = 4/5 = 80%
Recall = 4/10 = 40%
```

### Стратегии в зависимости от задачи

**Высокая точность (качество важнее количества):**
```javascript
const results = await search(query, k=3);
const filtered = results.filter(r => r.similarity > 0.7);
// Few results, but all highly relevant
```

**Высокая полнота (количество важнее качества):**
```javascript
const results = await search(query, k=50);
const filtered = results.filter(r => r.similarity > 0.3);
// Many results, some less relevant
```

**Сбалансированный подход:**
```javascript
const results = await search(query, k=10);
const filtered = results.filter(r => r.similarity > 0.5);
// Good mix of quality and quantity
```

### Переранжирование для повышения качества

**Двухэтапная выборка:**

```
Stage 1: Fast approximate search (HNSW)
  - Fetch k=100 candidates
  - Time: 10ms
  - Recall: 95%

Stage 2: Expensive re-ranking
  - Re-score with better model
  - Pick top k=5
  - Time: 40ms
  - Precision: 98%

Total: 50ms with excellent quality
```

**Когда использовать:**
- Качество критически важно
- Можно себе позволить дополнительную задержку
- Есть хорошая модель переранжирования

## Типичные паттерны и антипаттерны

### ✅ Хорошие паттерны

**1. Кэшируйте эмбеддинги для повторных запросов**
```javascript
const embedCache = new Map();
// Cache populated over time
```

**2. Используйте пакетные операции, когда это возможно**
```javascript
const results = await Promise.all(
    queries.map(q => search(q))
);
```

**3. Отслеживайте производительность отдельно**
```javascript
logMetrics({ embedTime, searchTime, totalTime });
// Identify true bottlenecks
```

**4. Избыточная выборка, затем фильтрация**
```javascript
const candidates = await search(query, k * 2);
const filtered = candidates.filter(condition).slice(0, k);
```

**5. Устанавливайте пороги релевантности**
```javascript
const results = await search(query, 20);
const relevant = results.filter(r => r.similarity > 0.5);
```

### ❌ Антипаттерны

**1. Повторная генерация эмбеддингов для одного и того же запроса**
```javascript
// Bad: Embed every time
for (let i = 0; i < 10; i++) {
    await search("same query", 5);  // Wastes 450ms!
}

// Good: Embed once
const emb = await embed("same query");
for (let i = 0; i < 10; i++) {
    await knnSearch(emb, 5);  // Only 50ms total
}
```

**2. Последовательная обработка**
```javascript
// Bad: One at a time
for (const q of queries) {
    await search(q);  // 200ms total
}

// Good: Parallel
await Promise.all(queries.map(search));  // 50ms total
```

**3. Неправильное значение k**
```javascript
// Bad: Too small
const results = await search(query, 1);
// Might miss relevant results

// Bad: Too large
const results = await search(query, 1000);
// Mostly noise

// Good: Balanced
const results = await search(query, 5);
```

**4. Игнорирование оценок сходства**
```javascript
// Bad: Return all k results
return await search(query, 10);

// Good: Filter by threshold
const results = await search(query, 10);
return results.filter(r => r.similarity > 0.5);
```

## Соображения масштабирования

### Малый масштаб (< 1 000 документов)

**Характеристики:**
- Любой алгоритм работает нормально
- Полный перебор приемлем
- Оптимизация направлена на скорость разработки

**Рекомендации:**
- Простое хранилище в оперативной памяти
- Не нужен сложный индекс
- Сфокусируйтесь на корректности, а не на производительности

### Средний масштаб (1 000 – 100 000 документов)

**Характеристики:**
- HNSW даёт явные преимущества
- Время поиска менее 100 мс
- Помещается в оперативную память одной машины

**Рекомендации:**
- Используйте HNSW (embedded-vector-db)
- Кэшируйте эмбеддинги
- Пакетные операции
- Отслеживайте производительность

### Большой масштаб (> 100 000 документов)

**Характеристики:**
- Необходим приближённый поиск
- Рассмотрите распределённые решения
- Возможно потребуется специализированное оборудование

**Рекомендации:**
- Распределённая векторная база данных (Qdrant, Weaviate)
- GPU для генерации эмбеддингов
- Продвинутые стратегии кэширования
- Горизонтальное масштабирование

## Сводка ключевых концепций

### 1. Поиск ближайших соседей

**Основная концепция:** Найти k наиболее похожих элементов на запрос.

**Почему это важно:** Основа всех операций векторного поиска.

### 2. k ближайших соседей

**Параметр k:** Управляет количеством возвращаемых результатов.

**Рекомендации:**
- k=3–5: Для общих задач
- k=10–20: При фильтрации или переранжировании
- k=50+: Требуется редко

### 3. Точный vs приближённый

**Точный (полный перебор):**
- Сложность O(N)
- 100% точность
- Подходит для < 1 000 документов

**Приближённый (HNSW):**
- Сложность O(log N)
- 95–99% точность
- Необходим для > 1 000 документов

### 4. Метрики расстояний

**Косинусное сходство:** Стандарт для семантического поиска (измеряет угол)

**Евклидово расстояние:** Альтернатива, когда величина имеет значение

### 5. Оптимизация производительности

**Ключевой вывод:** Генерация эмбеддингов является узким местом (90% времени)

**Лучшие практики:**
- Кэшируйте эмбеддинги
- Пакетные операции
- Предварительное вычисление, когда это возможно
- Отслеживайте отдельно

### Что мы узнали

**Поиск ближайших соседей — это:**
- ✅ Нахождение k наиболее похожих векторов на запрос
- ✅ Основа векторных баз данных
- ✅ Оптимизируемый с помощью алгоритма HNSW
- ✅ Ограниченный генерацией эмбеддингов, а не поиском

**Ключевые алгоритмы:**
1. **Полный перебор**: Простой, точный, O(N) — для малых наборов данных
2. **HNSW**: Быстрый, приближённый, O(log N) — для продакшна

**Оптимизация производительности:**
- Кэширование эмбеддингов (главный выигрыш)
- Пакетные операции (ускорение в 4 раза)
- Предварительное вычисление офлайн (огромное ускорение)
- Избыточная выборка для фильтрации

### Лучшие практики

1. **Используйте HNSW** для наборов данных > 1 000 документов
2. **Выбирайте k осознанно**: 5–10 для большинства задач
3. **Кэшируйте эмбеддинги** для повторных запросов
4. **Пакетные операции** с Promise.all()
5. **Отслеживайте производительность** отдельно (эмбеддинг vs поиск)
6. **Устанавливайте пороги** для фильтрации некачественных результатов
