# Создание хранилища векторов в памяти — обзор концепции

## Что такое векторное хранилище?

**Векторное хранилище** (также называемое векторной базой данных) — это специализированная база данных, предназначенная для эффективного хранения и поиска векторов высокой размерности.

Представьте его как **поисковую систему по смыслу**, которая находит похожие элементы на основе их значения, а не точного совпадения текста.

### Какую задачу оно решает

**Традиционный поиск по ключевым словам:**
```javascript
// Can only find exact matches or word variations
searchDatabase("Python programming");
// Finds: Documents containing "Python" AND/OR "programming"
// Misses: "Learn to code", "Beginner's guide to scripting"
```

**Семантический поиск на основе векторов:**
```javascript
// Finds semantically similar content
vectorStore.search(embed("Python programming"), k=5);
// Finds: 
// - "Python is a programming language..."
// - "JavaScript for web development..."
// - "Learn coding basics..."
// All are conceptually related!
```

## Зачем векторные хранилища важны для RAG

В системе **RAG (Retrieval-Augmented Generation, генерация с расширением поиска)**:

1. **Хранение знаний:** преобразование документов в эмбеддинги и их сохранение
2. **Эффективный поиск:** поиск релевантных документов по запросу пользователя
3. **Расширение возможностей LLM:** предоставление контекста языковой модели
4. **Генерация ответов:** LLM формирует обоснованные ответы

Без векторного хранилища системы RAG не могут эффективно искать по обширным коллекциям документов.

### Рабочий процесс RAG

```
Запрос пользователя
    ↓
Эмбеддинг запроса → [0.23, -0.45, 0.12, ...]
    ↓
Поиск в векторном хранилище (поиск по сходству)
    ↓
Получение top-K документов
    ↓
Объединение запроса + извлечённого контекста
    ↓
Отправка в LLM
    ↓
Генерация ответа
```

**Ключевой вывод:** векторное хранилище — это и есть «поиск» в схеме «генерация с расширением поиска»!

## Хранилище в памяти и постоянные векторные хранилища

### Векторные хранилища в памяти

**Что это:**  хранят все векторы исключительно в оперативной памяти без автоматической записи на диск.  
(Ручная персистентность доступна только в том случае, если разработчик явно добавляет логику `save()` / `load()`.)

**Характеристики:**
- ✅ Крайне быстрые (без ввода-вывода на диск)
- ✅ Простая настройка (без внешних сервисов)
- ✅ Идеальны для разработки
- ❌ Данные **теряются при перезапуске, если не сохранять вручную**
- ❌ Ограничены объёмом доступной оперативной памяти
- ❌ Не подходят для продакшена без логики персистентности

**Когда использовать:**
- Разработка и прототипирование
- Тестирование и оценка
- Небольшие наборы данных (< 10 000 документов)
- Кратковременные процессы
- Локальные эксперименты
- Сценарии, где персистентность не нужна или сохранение осуществляется вручную

### Постоянные векторные хранилища

**Что это:**  хранят векторы на диске (или в распределённых узлах) и загружают их обратно в память при необходимости.  
Персистентность является встроенной и автоматической или полуавтоматической в зависимости от базы данных.

**Примеры:**
- **Qdrant**
- **Neo4j**
- **Chroma**
- **Pinecone**
- **LanceDB**
- **Weaviate**

**Характеристики:**
- ✅ Данные сохраняются между перезапусками
- ✅ Масштабирование до миллионов векторов
- ✅ Готовность к продакшену
- ✅ Часто распределённые и отказоустойчивые
- ❌ Более сложная настройка
- ❌ Требуется запущенный сервер или уровень хранения
- ❌ Немного медленнее чистого хранилища в памяти (из-за работы с диском)

**Когда использовать:**
- Продуктивные приложения
- Большие наборы данных (> 10 000 документов)
- Необходимость в автоматической и надёжной персистентности
- Много пользовательские приложения
- Долгоживущие микросервисы
- Системы, требующие резервного копирования, репликации или версионирования

## Как работает векторный поиск

### Шаг 1: Создание эмбеддингов

**Эмбеддинг:** числовое представление текста в виде вектора высокой размерности.

```javascript
"Python programming" → [0.23, -0.45, 0.12, 0.87, -0.34, ...]
                       (384 dimensions for BGE-small)
```

**Ключевое свойство:** похожие концепции имеют похожие эмбеддинги.

```javascript
embed("dog") ≈ embed("puppy")  // Similar vectors
embed("dog") ≠ embed("car")    // Different vectors
```

### Шаг 2: Хранение векторов

**Что хранится:**
- **Вектор**: эмбеддинг (например, 384 числа)
- **ID**: уникальный идентификатор
- **Метаданные**: исходное содержание и дополнительные данные

```javascript
{
    id: "doc_1",
    vector: [0.23, -0.45, ...],  // 384 dimensions
    metadata: {
        content: "Python is a programming language",
        category: "programming",
        author: "John Doe"
    }
}
```

### Шаг 3: Поиск по сходству

**Цель:** найти векторы, наиболее похожие на вектор запроса.

**Метод:** косинусное сходство (измеряет угол между векторами)

```javascript
// Query
queryVector = embed("Learn coding");

// Compute similarity with each stored vector
similarity(queryVector, doc1.vector) = 0.82  // High similarity
similarity(queryVector, doc2.vector) = 0.65
similarity(queryVector, doc3.vector) = 0.43  // Low similarity

// Return top K results
topK = [doc1, doc2, doc3]
```

**Косинусное сходство:**
- Диапазон: от -1 до 1 (или преобразуется в диапазон от 0 до 1)
- 1 = идентичное направление
- 0 = ортогональные (не связаны)
- -1 = противоположное направление

### Шаг 4: Фильтрация и ранжирование

**Фильтрация (опционально):**
```javascript
// Only return documents matching metadata criteria
results = search(query, k=10, filter={
    category: "programming",
    difficulty: "beginner"
});
```

**Ранжирование результатов:**
- Результаты сортируются по оценке сходства (от наиболее к наименее)
- Применяется порог для отсеивания несоответствующих результатов
- При необходимости повторное ранжирование с использованием более сложных моделей

## Ключевые концепции

### 1. Эмбеддинги: основа

**Что делает эмбеддинги хорошими:**
- **Захват семантического смысла:** похожие концепции группируются вместе
- **Единая размерность:** все векторы одного размера
- **Соответствие домену:** модель обучена на релевантных данных

**Популярные модели эмбеддингов:**
- **BGE-small (384D)**: быстрая, хороша для общего использования
- **BGE-large (1024D)**: более высокое качество, медленнее
- **OpenAI text-embedding-3-small**: облачная
- **Sentence-Transformers**: множество специализированных моделей

**Ключевое правило:** используйте **одну и ту же модель эмбеддингов** для документов и запросов!

### 2. HNSW: алгоритм поиска

**Иерархический навигируемый малый мир (HNSW):**
- Графовый поиск приближённого ближайшего соседа (ANN)
- Очень быстрый: время поиска O(log N)
- Высокая полнота: находит наиболее релевантные результаты
- Компромисс: приближённый, а не точный

**Почему HNSW:**
- Точный поиск слишком медленен для больших наборов данных: O(N)
- HNSW примерно в 100 раз быстрее при минимальной потере точности
- Отраслевой стандарт для векторного поиска

**Как это работает (упрощённо):**
1. Построение многослойного графа векторов
2. Навигация от точки входа к запросу
3. Использование свойства «малого мира» для пропуска нерелевантных областей
4. Возврат приближённых ближайших соседей

### 3. Пространства имён: изоляция данных

**Что это:** логические разделы внутри одной базы данных.

```javascript
// Different collections, same database
await db.insert("products", "p1", vector, {...});
await db.insert("articles", "a1", vector, {...});
await db.insert("users", "u1", vector, {...});

// Search only products
await db.search("products", queryVector, 5);
```

**Преимущества:**
- Организация различных типов данных
- Более быстрый поиск (меньшее пространство поиска)
- Упрощение управления и очистки
- Многопользовательские приложения

### 4. Метаданные: добавление контекста

**Помимо векторов:** метаданные позволяют хранить структурированную информацию наряду с эмбеддингами.

```javascript
{
    vector: [...],
    metadata: {
        content: "The document text",
        title: "Document Title",
        author: "John Doe",
        date: "2024-01-15",
        category: "technology",
        tags: ["AI", "machine learning"],
        source: "blog.example.com",
        language: "en"
    }
}
```

**Сценарии использования:**
- **Фильтрация**: показывать только документы за 2024 год
- **Отображение**: показывать заголовок и автора пользователю
- **Бизнес-логика**: маршрутизация по категории или источнику
- **Аналитика**: отслеживание популярных тем или авторов

## Пакет `embedded-vector-db`

### Что это

**Пакет:** `embedded-vector-db` (npm)
- **Статус:** бета (API может измениться до версии 1.0)
- **Назначение:** встроенная векторная база данных для Node.js
- **Основа:** hnswlib-node для поиска kNN
- **Лицензия:** MIT

### Почему мы его используем

**Преимущества:**
1. **Чистый JavaScript/TypeScript**: отдельный сервер базы данных не нужен
2. **Встроенный**: работает в вашем процессе Node.js
3. **Простой API**: легко изучить и использовать
4. **Быстрый**: поиск на основе HNSW
5. **Лёгкий**: минимальное количество зависимостей

**Идеально подходит для:**
- Изучения и экспериментов
- Небольших и средних проектов
- Прототипирования систем RAG
- Разработки и тестирования

### Основной API

```javascript
// Initialize
const db = new VectorDB({
    dim: 384,           // Vector dimensions
    maxElements: 10000  // Maximum capacity
});

// Create (Insert)
await db.insert(namespace, id, vector, metadata);

// Read (Search)
const results = await db.search(namespace, queryVector, k);

// Read (Get by ID)
const doc = await db.get(namespace, id);

// Update
await db.update(namespace, id, newVector, newMetadata);

// Delete
await db.delete(namespace, id);
```

**Простые операции CRUD** — точно так же, как в традиционных базах данных!

### Поток данных

**Добавление документов:**
1. Ваше приложение предоставляет текстовые документы
2. Модель эмбеддингов преобразует текст в векторы
3. Вспомогательная функция вызывает `VectorDB.insert()`
4. `VectorDB` сохраняет в индекс HNSW
5. Метаданные хранятся рядом с векторами

**Поиск:**
1. Пользователь вводит текст запроса
2. Модель эмбеддингов преобразует запрос в вектор
3. Вспомогательная функция вызывает `VectorDB.search()`
4. HNSW находит k ближайших соседей
5. Результаты возвращаются с метаданными
6. Ваше приложение отображает или обрабатывает результаты

## Сценарии использования в реальном мире

### Сценарий 1: Поиск по документации

**Задача:** создать семантический поиск по вашей документации.

**Реализация:**
1. Разделить документацию на фрагменты
2. Сгенерировать эмбеддинги для каждого фрагмента
3. Сохранить в векторной базе данных с метаданными (заголовок, URL, раздел)
4. Пользователь ищет: «Как мне развернуть?»
5. Векторный поиск находит релевантные документы
6. Отображение первых результатов со ссылками

**Преимущества:**
- Находит релевантные документы даже без точных ключевых слов
- Лучше, чем Ctrl+F или традиционный поиск
- Работает с несколькими источниками документации

### Сценарий 2: Бот для поддержки клиентов

**Задача:** отвечать на вопросы клиентов, используя прошлые обращения.

**Реализация:**
1. Эмбеддинг всех решённых обращений в поддержку
2. Сохранение с метаданными (категория, решение, удовлетворённость)
3. Пользователь задаёт вопрос
4. Поиск похожих прошлых обращений
5. Извлечение решений
6. LLM генерирует ответ на основе похожих обращений

**Преимущества:**
- Использование накопленных знаний
- Единообразие ответов
- Снижение нагрузки на команду поддержки

### Сценарий 3: Поиск по коду

**Задача:** находить похожие фрагменты кода в большом репозитории.

**Реализация:**
1. Эмбеддинг функций, классов или файлов
2. Сохранение с метаданными (путь к файлу, язык, автор)
3. Разработчик ищет: «логика аутентификации»
4. Поиск семантически похожего кода
5. Показ примеров из репозитория

**Преимущества:**
- Обнаружение паттернов по всему репозиторию
- Содействие повторному использованию кода
- Быстрое обучение новых разработчиков

### Сценарий 4: Система рекомендаций

**Задача:** рекомендовать похожие статьи или товары.

**Реализация:**
1. Эмбеддинг статей/товаров
2. Пользователь просматривает элемент
3. Поиск похожих эмбеддингов
4. Рекомендация top-K результатов
5. Фильтрация по категории или предпочтениям пользователя

**Преимущества:**
- Рекомендации на основе содержания
- Работает даже для новых элементов (прлема холодного старта)
- Изначально не требуется коллаборативная фильтрация

## Характеристики производительности

### Сравнение скорости

**Точный поиск (полный перебор):**
- Сравнение запроса с каждым вектором
- Время: O(N × D), где N = количество документов, D = размерность
- Пример: 10 000 документов × 384 измерения = 3,84 млн операций
- Слишком медленно для поиска в реальном времени!

**Приближённый поиск HNSW:**
- Навигация по графовой структуре
- Время: O(log N × D)
- Пример: log₂(10 000) × 384 ≈ 5 000 операций
- Примерно в 100 раз быстрее!

### Использование памяти

**На документ:**
```
Vector: 384 dimensions × 4 bytes (float32) = 1,536 bytes
Metadata: ~500-2000 bytes (JSON)
HNSW overhead: ~200 bytes
Total: ~2-4 KB per document
```

**Примеры расчётов:**
- 1 000 документов: ~2–4 МБ
- 10 000 документов: ~20–40 МБ
- 100 000 документов: ~200–400 МБ

**Практические ограничения:**
- В памяти: до ~1 млн документов на типичном компьютере
- Ограничено объёмом оперативной памяти
- Производительность снижается с ростом объёма

### Время эмбеддингов

**Узкое место:** генерация эмбеддингов — самая медленная часть.

**Типичное время (BGE-small на CPU):**
- Короткий текст (50 слов): ~40 мс
- Средний текст (200 слов): ~60 мс
- Длинный текст (500 слов): ~80 мс

**Стратегии оптимизации:**
- Пакетная генерация эмбеддингов, когда это возможно
- Использование ускорения на GPU
- Кэширование эмбеддингов
- Предварительное вычисление эмбеддингов в офлайне

## Типовые паттерны

### Паттерн 1: Семантический поиск с фильтрацией

**Цель:** найти релевантные документы в пределах категории.

```javascript
// Search with high k
const results = await vectorStore.search(namespace, queryVector, 20);

// Filter by metadata
const filtered = results
    .filter(r => r.metadata.category === "programming")
    .filter(r => r.metadata.difficulty === "beginner")
    .filter(r => r.score > 0.5)  // Relevance threshold
    .slice(0, 5);  // Top 5
```

### Паттерн 2: Гибридный поиск

**Цель:** объединить семантический и ключевой поиск.

```javascript
// Semantic search
const semanticResults = await vectorStore.search(ns, queryVector, 10);

// Keyword search (if supported)
const keywordResults = await vectorStore.fullTextSearch(ns, query, 10);

// Merge and deduplicate
const combined = mergeResults(semanticResults, keywordResults);
```

### Паттерн 3: Повторное ранжирование

**Цель:** улучшить качество поиска с помощью второй модели.

```javascript
// Initial search (fast, broad)
const candidates = await vectorStore.search(ns, queryVector, 20);

// Re-rank with better model (slower, more accurate)
const reranked = await reranker.rank(query, candidates);

// Return top results
return reranked.slice(0, 5);
```

### Паттерн 4: Фрагментация документов

**Цель:** работа с длинными документами путём разбиения на фрагменты.

```javascript
// Split document
const chunks = splitIntoChunks(longDocument, maxChunkSize=500);

// Embed each chunk
for (const [i, chunk] of chunks.entries()) {
    const embedding = await embed(chunk);
    await vectorStore.insert(ns, `${docId}_chunk_${i}`, embedding, {
        content: chunk,
        documentId: docId,
        chunkIndex: i,
        title: documentTitle
    });
}

// Search returns chunks, group by document
const results = await vectorStore.search(ns, queryVector, 10);
const groupedByDoc = groupBy(results, r => r.metadata.documentId);
```

## Рекомендации по использованию

### 1. Разделяйте документы на фрагменты надлежащим образом

**Проблема:** длинные документы создают шумовые эмбеддинги.

**Решение:**
- Разделяйте на семантические фрагменты (абзацы, разделы)
- Типичный размер: 200–500 слов
- Включайте контекст из окружающих фрагментов

### 2. Сохраняйте исходное содержание

**Всегда включайте содержание в метаданные:**
```javascript
await vectorStore.insert(ns, id, vector, {
    content: originalText,  // Essential!
    // ... other metadata
});
```

**Почему:** оно понадобится вам для отображения результатов и предоставления контекста LLM.

### 3. Используйте единые эмбеддинги

**Критическое правило:** одна модель для индексации и поиска.

```javascript
// ❌ Wrong
const docEmbedding = await modelA.embed(document);
const queryEmbedding = await modelB.embed(query);  // Different model!

// ✅ Correct
const docEmbedding = await model.embed(document);
const queryEmbedding = await model.embed(query);  // Same model
```

### 4. Устанавливайте пороги релевантности

**Фильтрация некачественных результатов:**
```javascript
const THRESHOLD = 0.5;
const relevantResults = results.filter(r => r.score > THRESHOLD);
```

**Настройка порога на основе:**
- Требований вашего сценария использования
- Характеристик модели
- Обратной связи от пользователей

### 5. Включайте богатые метаданные

```javascript
    tags: [...],
    source: "...",

}
```

**Ключевые метрики:**
- Время генерации эмбеддингов

## Путь миграции в продакшен

### Этап 1: Разработка (текущий)

**Инструменты:** `embedded-vector-db` (в памяти)

**Характеристики:**
- Быстрая итерация
- Отсутствие внешних зависимостей
- Идеально для обучения

### Этап 2: Локальная персистентность

**Инструменты:** `embedded-vector-db` с функциями save/load или LanceDB

**Реализация:**
```javascript
// Save to disk
await vectorStore.save(namespace, "./data/vectors");

// Load on restart
await vectorStore.load(namespace, "./data/vectors");
```

**Преимущества:**
- Данные сохраняются между перезапусками
- Всё ещё простая настройка
- Подходит для развёртывания на одном сервере

### Этап 3: Продакшен-масштаб

**Инструменты:** Qdrant, Weaviate, Chroma или Pinecone

**Шаги миграции:**
1. Экспорт векторов и метаданных из embedded-vector-db
2. Импорт в продакшен-хранилище векторов
3. Обновление кода поиска (минимальные изменения, если API аналогичен)
4. Развёртывание с надлежащим мониторингом

**Новые возможности:**
- Распределённая архитектура
- Высокая доступность
- Расширенные функции (фильтрация, гибридный поиск)
- Лучшая производительность в масштабе
