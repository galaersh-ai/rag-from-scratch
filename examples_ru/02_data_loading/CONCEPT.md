## Введение в загрузку документов

### Что такое загрузка документов?

Загрузка документов — это **первый и фундаментальный шаг** в любой системе Retrieval-Augmented Generation (RAG). Это процесс преобразования неструктурированных данных (PDF-файлов, веб-страниц, документов) в структурированный, машиночитаемый формат, который может быть обработан компонентами конвейера RAG.

Представьте это как «систему приёма» вашего приложения RAG — точно так же, как ресторану необходимо правильно получать, проверять и подготавливать ингредиенты перед приготовлением, система RAG должна правильно загружать и структурировать документы, прежде чем они смогут использоваться для ответа на вопросы.

### Почему загрузка документов важна

Качество загрузки документов напрямую влияет на:

1. **Точность поиска**: Неудачное извлечение текста приводит к неполному или повреждённому содержимому
2. **Качество эмбеддингов**: Мусор на входе — мусор на выходе; беспорядочный текст создаёт некачественные эмбеддинги
3. **Качество ответов**: LLM могут работать только с тем, что им предоставлено
4. **Точность цитирования**: Корректные метаданные обеспечивают точную атрибуцию источников
5. **Производительность системы**: Эффективная загрузка влияет на скорость всего конвейера

### Проблема

PDF-файлы и другие документы содержат:
- Визуальное форматирование (жирный шрифт, курсив, колонки)
- Нетекстовые элементы (изображения, диаграммы, таблицы)
- Структурные артефакты (заголовки, нижние колонтитулы, номера страниц)
- Сложности кодировки (шрифты, наборы символов)
- Информацию о макете (колонки, текстовые блоки)

Загрузчики должны извлекать осмысленный текст, разумно обрабатывая все эти сложности.

```

├─────────────────────────────────────────────────────────────┤

│                                                             │
│  3. ГЕНЕРАЦИЯ ЭМБЕДДИНГОВ                                   │

│                                                             │
│  4. ХРАНЕНИЕ ВЕКТОРОВ                                       │

│                                                             │
│  5. ПОИСК (во время запроса)                                │
│     ├─ Эмбеддинг пользовательского запроса                  │

│                                                             │
```

### Поток данных

```
PDF-файл → Загрузчик → Структурированные документы → Разбиватель текста → Чанки
                                      ↓
                              (с метаданными)
                                      ↓
                          В итоге становится контекстом для LLM
```

## Основные концепции

### 1. Абстракция документа

**Документ** — это фундаментальная абстракция в системах RAG. Он представляет собой единицу текста с привязанными метаданными:

```javascript
{
    pageContent: "The actual text content...",
    metadata: {
        source: "where it came from",
        page: "which page",
        // ... other contextual info
    }
}
```

**Почему это важно**:
- **Разделение ответственности**: Содержимое отделено от метаданных
- **Гибкость**: Одна и та же структура подходит для PDF, веб-страниц, CSV и т.д.
- **Отслеживаемость**: Всегда можно узнать происхождение информации
- **Фильтрация**: Можно фильтровать поиск по метаданным (например, «искать только в свежих документах»)

### 2. Преобразование неструктурированных → структурированных данных

Загрузка документов выполняет преобразование:

```
НЕСТРУКТУРИРОВАННЫЕ          СТРУКТУРИРОВАННЫЕ
┌──────────────┐          ┌──────────────┐
│ Бинарный PDF │          │ Document {   │
│ со сложным   │   →→→    │   text: "...",│
│ форматиров.  │          │   metadata   │
└──────────────┘          └──────────────┘
```

Это преобразование включает:
- **Парсинг**: понимание формата файла
- **Извлечение**: извлечение текстового содержимого
- **Очистка**: удаление артефактов
- **Структурирование**: организация в стандартный формат

### 3. Метаданные как первоэлемент

Метаданные — это не просто «дополнительная информация» — они критически важны для:

#### Атрибуция источника

- Пользователь спрашивает: «Что говорится в статье о RAG?»
- LLM может ответить: «Согласно пятой странице статьи…»
- Это возможно только потому, что мы сохранили номера страниц!

#### Фильтрация и маршрутизация

- «Искать только в документах за 2024 год»
- «Найти информацию в отчётах за третий квартал»
- Метаданные позволяют выполнять такие запросы

#### Отладка и аудит

- «Откуда взялся этот ответ?»
- Метаданные предоставляют обратный путь к источнику


### 4. Решения о гранулярности

**Гранулярность** определяет, насколько мелко вы разбиваете документы:

```
ГРУБАЯ ГРАНУЛЯРНОСТЬ        ТОНКАЯ ГРАНУЛЯРНОСТЬ
┌──────────────┐            ┌───┬───┬───┬───┐
│              │            │ P │ P │ P │ P │
│  Полный      │     vs.    │ a │ a │ a │ a │
│  документ    │            │ g │ g │ g │ g │
│              │            │ e │ e │ e │ e │
└──────────────┘            └───┴───┴───┴───┘
```

Компромиссы:
- **Грубая**: Лучший контекст, но менее точный поиск
- **Тонкая**: Более точный поиск, но может потерять контекст

## Структура документа

### Философия класса Document

```javascript
class Document {
    constructor(pageContent, metadata, id) {
        this.pageContent = pageContent;
        this.metadata = metadata;
        this.id = id;
    }
}
```

Эта простая структура воплощает несколько принципов проектирования:

#### 1. Простота
- Три поля, ясное назначение
- Без лишней сложности
- Легко понять и использовать

#### 2. Гибкость
- `pageContent` может содержать любой текст
- `metadata` — открытый объект (любая структура)
- Работает для разных типов документов

#### 3. Совместимость
- Соответствует интерфейсу Document из LangChain
- Взаимодействует с другими инструментами RAG
- Стандартный паттерн в экосистеме

#### 4. Расширяемость
- Легко наследуется для специфических типов документов
- Метаданные могут хранить доменную информацию
- Идентификатор позволяет отслеживать и устранять дубликаты

### Проектирование схемы метаданных

Структура метаданных в нашем коде:

```javascript
{
    source: "URL or file path",           // Origin
    pdf: {
        info: {...},                       // PDF properties
        metadata: {...},                   // PDF metadata
        totalPages: 22                       // Total pages
    },
    loc: {                                 // Location
        pageNumber: 1                      // Specific page
    }
}
```

**Соображения при проектировании**:

**Иерархическая организация**:
- Группирует связанные метаданные вместе
- `pdf` содержит всю информацию, специфичную для PDF
- Легко расширяется для других форматов (добавить `word: {...}`)

**Баланс между универсальностью и специфичностью**:
- `source` и `loc` — универсальные (работают для любого документа)
- `pdf` — специфичные для формата (только для PDF)

**Адаптированность к будущему**:
- Можно добавлять новые ключи верхнего уровня без нарушения существующего кода
- Вложенная структура позволяет глубокую настройку

## Стратегии извлечения текста

### Проблема извлечения текста из PDF

PDF — это по своей природе **визуальные документы** — они указывают, где должен появиться чернильный знак на странице, а не логическую структуру текста. Это создаёт проблемы:

#### Проблема 1: Неоднозначность порядка текста

```
┌─────────────────────────────┐
│  Визуальный макет PDF:      │
│                             │
│  ┌──────────┐  ┌──────────┐ │
│  │ Колонка 1│  │ Колонка 2│ │
│  │ Текст A  │  │ Текст C  │ │
│  │ Текст B  │  │ Текст D  │ │
│  └──────────┘  └──────────┘ │
└─────────────────────────────┘

Возможные варианты извлечения:
1. "Text A, Text B, Text C, Text D" (последовательно по колонкам)
2. "Text A, Text C, Text B, Text D" (слева направо, сверху вниз)
```

#### Проблема 2: Загрязнение артефактами

PDF-файлы содержат визуальные элементы, которые не должны попадать в извлекаемый текст:
- Номера страниц («Стр. 1 из 22»)
- Заголовки и нижние колонтитулы
- Водяные знаки
- Декоративные элементы

#### Проблема 3: Проблемы кодировки

- Разные шрифты могут кодировать символы по-разному
- Специальные символы (маркеры, длинные тире) могут не извлекаться корректно
- Проблемы нормализации Unicode

### Философия извлечения

Наш код использует **pdf-parse**, который:
- Извлекает текст в порядке чтения (лучшая попытка)
- Обрабатывает типичные проблемы кодировки
- Предоставляет извлечение на уровне страниц

Философия:
1. **Сначала извлечь, потом очистить**: Получить весь текст, а затем его обработать
2. **Сохранять семантическую структуру**: Сохранять разрывы абзацев
3. **Удалять шум**: Убирать артефакты нумерации страниц
4. **Нормализовать**: Создать единообразное форматирование

## Философия предобработки текста

### Зачем нужна очистка текста?

Необработанный текст из PDF выглядит примерно так:

```
"Retrieval-Augmented    Generation  for


AI-Generated Content

-- 1 of 22 --


Abstract:    This paper surveys..."
```

После очистки:

```
"Retrieval-Augmented Generation for
AI-Generated Content

Abstract: This paper surveys..."
```

### Конвейер очистки

Наша функция `cleanText` реализует **трёхэтапный конвейер очистки**:

#### Этап 1: Удаление артефактов
```javascript
.replace(/[-–-]\s*\d+\s*of\s*\d+\s*[-–-]/gi, '')
```

**Назначение**: Удаление артефактов нумерации страниц, которые не несут семантической ценности

**Философия**: Эти элементы являются *навигационными подсказками* для людей, просматривающих PDF, но они являются шумом для машинной обработки. Они:
- Не вносят вклад в смысл
- Искусственно разбивают предложения
- Напрасно потребляют токены
- Вводят в заблуждение модели эмбеддингов

**Пример**:
```
До: "The study found -- 5 of 12 -- significant results"
После: "The study found significant results"
```

#### Этап 2: Нормализация пробелов
```javascript
.replace(/[ \t]+/g, ' ')
```

**Назначение**: Обеспечить единообразные пробелы

**Философия**: Множественные пробелы и табуляции — это артефакты рендеринга PDF. Они не несут семантического смысла в обычном тексте. Нормализация:
- Делает текст предсказуемым
- Уменьшает количество токенов
- Улучшает согласованность эмбеддингов
- Упрощает последующую обработку

**Пример**:
```
До: "The    results     showed\tsignificant"
После: "The results showed significant"
```

#### Этап 3: Сохранение абзацев
```javascript
.replace(/\n{3,}/g, '\n\n')
```

**Назначение**: Сохранение структуры документа при удалении лишнего

**Философия**: Это самый тонкий, но важный шаг. Он уравновешивает два конкурирующих требования:

1. **Сохранение структуры**: Разрывы абзацев несут семантический смысл (смена темы, логические разделы)
2. **Удаление шума**: Множественные пустые строки — это артефакты рендеринга

**Почему два переноса?**:
- Стандартная конвенция: `\n\n` = разрыв абзаца
- Один перенос = перенос строки (внутри абзаца)
- Два переноса = граница абзаца
- Три и более переноса = артефакт рендеринга

**Пример**:
```
До:
"First paragraph.


Second paragraph."

После:
"First paragraph.

Second paragraph."
```

### Философия консервативной очистки

Обратите внимание, что наша очистка **не делает**:
- ❌ Удаляет пунктуацию
- ❌ Переводит всё в нижний регистр
- ❌ Удаляет стоп-слова
- ❌ Выполняет лемматизацию или стемминг
- ❌ Исправляет орфографические ошибки

**Почему?** Потому что:
1. Современные модели эмбеддингов справляются с такими вариациями
2. Сохраняется первоначальный смысл и авторский стиль
3. Сохраняются имена собственные и технические термины
4. Уменьшается риск внесения ошибок
5. Сохраняется точность цитат

Это **консервативная предобработка** — удаляем очевидный шум, сохраняем смысл.

## Разбиение по страницам vs. полный документ

### Основной компромисс

Это одно из наиболее важных концептуальных решений при загрузке документов:

```
┌────────────────────────────────────────────────────────┐
│              СПЕКТР ГРАНУЛЯРНОСТИ                      │
├────────────────────────────────────────────────────────┤
│                                                        │
│  Полный документ       Страницы          Абзацы        │
│  ───────────────       ───────           ─────────      │
│  • Лучший контекст     • Сбалансиров.   • Лучшая       │
│  • Худшая точность     • Цитаты         • Точность     │
│  • Простота            • Гибкость       • Сложность    │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### Режим полного документа (`splitPages: false`)

**Концепция**: Обрабатывать весь PDF как единый монолитный документ.

**Когда использовать**:
- Короткие документы (< 10 страниц)
- Высокосвязное содержимое (одна тема на протяжении всего документа)
- Когда вы планируете выполнять собственное разбиение на чанки позже
- Простые сценарии использования

**Преимущества**:
```
✓ Максимальное сохранение контекста
✓ Отсутствие искусственных границ
✓ Более простая структура данных (1 документ)
✓ Более быстрая обработка (однократное извлечение)
✓ Подходит для вопросов по всему документу
```

**Недостатки**:
```
✗ Может превысить лимит токенов модели эмбеддингов
✗ Менее точный поиск
✗ Отсутствие цитат на уровне страниц
✗ Сложнее выделить релевантные разделы
✗ Потребление памяти для крупных документов
```

**Пример сценария использования**:
```
Документ: 5-страничная политика компании по удалённой работе
Стратегия: Загрузить как единый документ
Причина: Короткий, связный, пользователям нужен полный контекст
```

### Режим разбиения по страницам (`splitPages: true`)

**Концепция**: Создавать один документ на каждую страницу, каждый со своими метаданными.

**Когда использовать**:
- Длинные документы (> 10 страниц)
- Академические статьи (важны цитаты)
- Технические руководства (конкретные ссылки на страницы)
- Когда нужен гранулярный поиск

**Преимущества**:
```
✓ Точные цитаты на уровне страниц
✓ Управляемые размеры чанков
✓ Лучшая гранулярность поиска
✓ Возможна параллельная обработка
✓ Уважение естественных границ
```

**Недостатки**:
```
✗ Контекст может разбиваться по страницам
✗ Больше документов для управления
✗ Немного более медленная обработка
✗ Нужна хорошая стратегия разбиения на чанки в дальнейшем
```

**Пример сценария использования**:
```
Документ: 50-страничная научная статья
Стратегия: Разбить по страницам
Причина: Нужны цитаты, документ длинный, темы различаются по разделам
```

### Золотая середина

На практике страницы часто всё ещё слишком велики для оптимального поиска. Типичный рабочий процесс:

```
PDF → Загрузка по страницам → Дальнейшее разбиение на чанки → Эмбеддинг

Пример:
1. Загрузка: 50-страничный PDF → 50 документов-страниц
2. Разбиение: Каждая страница → 2-4 чанка (семантические границы)
3. Результат: ~150 чанков, каждый с метаданными номера страницы
```

Это даёт вам:
- Лучшее из обоих миров
- Атрибуцию на уровне страниц
- Точность на уровне чанков
- Сохранение контекста за счёт перекрытия

## Сравнение с LangChain.js

### Обзор

LangChain.js — ведущий фреймворк для создания приложений на LLM, включая системы RAG. Сравним нашу собственную реализацию с подходом LangChain.

### Загрузчики документов в LangChain

LangChain предоставляет богатую экосистему загрузчиков документов:

```javascript
import { PDFLoader } from "langchain/document-loaders/fs/pdf";
import { WebPDFLoader } from "langchain/document-loaders/web/pdf";

// URL-based loading
const loader = new WebPDFLoader("https://example.com/paper.pdf");

// File-based loading
const loader = new PDFLoader("./path/to/file.pdf");

// Load documents

```

### Архитектурное сравнение

#### Наша реализация

**Философия**:
- **Явное управление** каждым шагом

- **Ориентация на обучение** (понимание процесса)

- **Абстракция** скрывает сложность

## Заключение

### Ключевые выводы

1. **Загрузка документов является фундаментальной**: Качество на этом этапе влияет на всё последующее

2. **Простое зачастую лучше**: Наша реализация доказывает, что для эффективности не нужна сложность

3. **Понимайте компромиссы**: Каждое решение проектирования имеет свои плюсы и минусы

4. **Начинайте просто, эволюционируйте**: Начните с базовой загрузки, добавляйте сложность по мере необходимости

5. **Выбирайте правильный инструмент**: Собственная реализация для обучения, LangChain для продакшна, гибрид для особых задач

### Путь

```
Этап 1: Изучение концепций (собственная реализация)
    ↓
Этап 2: Понимание сложностей (столкновение с пограничными случаями)
    ↓
Этап 3: Оценка фреймворков (понимание, зачем они существуют)
    ↓
Этап 4: Обоснованный выбор (собственная реализация vs. фреймворк)
```

### Дальнейшее изучение

Для углубления понимания:

1. **Экспериментируйте**: Попробуйте разные стратегии очистки, сравните результаты
2. **Измеряйте**: Добавьте метрики, посмотрите, что влияет на качество
3. **Сравнивайте**: Загружайте один и тот же документ с помощью собственного кода и LangChain
4. **Расширяйте**: Добавьте извлечение таблиц, обработку изображений
5. **Масштабируйте**: Попробуйте загрузить 100 документов, измерьте производительность

### Заключительные мысли

Загрузка документов — это точка пересечения инженерии данных и информационного поиска. Это:

Лучшие системы RAG не просто загруют документы — они **управляют знаниями**. И всё начинается здесь, на самом первом шаге, с обдуманной, принципиальной загрузки документов.
