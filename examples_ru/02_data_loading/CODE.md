# Документация по коду: извлечение текста из PDF для RAG

## Обзор

Этот модуль предоставляет функциональность для извлечения и предобработки текста из PDF-документов, специально разработанную для приложений Retrieval-Augmented Generation (RAG). Код выполняет разбор PDF, очистку текста и структурирование документов для подготовки содержимого к обработке в конвейерах RAG.

**Примечание:** это очень упрощённый пример. В реальных приложениях RAG данные обычно поступают из множества разнородных источников (базы данных, API, веб-страницы, документы в различных форматах и т. д.), а не только из PDF.

## Контекст: RAG (Retrieval-Augmented Generation)

Код является частью реализации RAG — техники, которая улучшает ответы больших языковых моделей (LLM) за счёт поиска релевантной информации во внешних документах. Ссылки на статьи в конце кода представляют собой фундаментальные исследования в области RAG:

- **Retrieval-Augmented Generation for AI-Generated Content**: обзор техник RAG
- **A Comprehensive Survey of RAG**: эволюция и текущий ландшафт
- **Corrective Retrieval Augmented Generation**: продвинутый RAG с механизмами коррекции

```javascript
import {PDFParse} from 'pdf-parse';
```

Код использует библиотеку `pdf-parse`, которая предоставляет возможности разбора PDF для приложений Node.js.

## Основные компоненты

### 1. Класс Document

```javascript
class Document {
    constructor(pageContent, metadata, id) {
        this.pageContent = pageContent;
        this.metadata = metadata;
        this.id = id;
    }
}
```

**Назначение**: представляет структурированный объект документа, инкапсулирующий извлечённый текст и связанные метаданные.

**Свойства**:
- `pageContent`: очищённое текстовое содержимое, извлечённое из PDF
- `metadata`: объект с информацией о документе (URL-источник, метаданные PDF, номера страниц)
- `id`: необязательный идентификатор документа (в текущей реализации устанавливается как `undefined`)

**Применение в RAG**: такая структура обычно используется в векторных базах данных и фреймворках RAG (например, LangChain) для хранения документов вместе с их эмбеддингами и метаданными для последующего поиска.

### 2. Функция cleanText

```javascript
const cleanText = (text) => {
    return text
        .replace(/[-–—]\s*\d+\s*of\s*\d+\s*[-–—]/gi, '')
        .replace(/[ \t]+/g, ' ')
        .replace(/\n{3,}/g, '\n\n')
        .trim();
}
```

**Назначение**: предобрабатывает сырой текст PDF для удаления артефактов и нормализации форматирования.

**Этапы очистки**:

1. **Удаление маркеров страниц**:
   ```javascript
   .replace(/[-–—]\s*\d+\s*of\s*\d+\s*[-–—]/gi, '')
   ```
    - Удаляет маркировку страниц вида \"-- 1 of 22 --\" или \"— 5 of 12 —\"
    - Использует регулярное выражение для распознавания различных типов тире (дефис, среднее тире, длинное тире)
    - Флаги `gi`: глобальный поиск (все вхождения) и нечувствительность к регистру

2. **Нормализация пробелов**:
   ```javascript
   .replace(/[ \t]+/g, ' ')
   ```
    - Схлопывает несколько последовательных пробелов и табуляций в один пробел
    - Улучшает согласованность текста для генерации эмбеддингов

3. **Сохранение структуры абзацев**:
   ```javascript
   .replace(/\n{3,}/g, '\n\n')
   ```
    - Сокращает 3 и более последовательных перевода строки ровно до 2
    - Сохраняет разрывы абзацев при удалении избыточных отступов
    - Важно для семантического разбиения на блоки в RAG

4. **Обрезка краёв**:
   ```javascript
   .trim()
   ```
    - Удаляет пробелы в начале и в конце строки

**Почему это важно для RAG**: чистый текст улучшает:
- качество эмбеддингов (векторных представлений)
- определение границ блоков
- эффективность токенов при отправке в LLM
- релевантность поиска при извлечении

### 3. Функция constructDocument

```javascript
const constructDocument = (pageContent, source, info, metadata, numPages, pageNumber) => {
    return new Document(
        pageContent,
        {
            source,
            pdf: {
                info,
                metadata,
                numPages,
            },
            loc: {
                pageNumber
            }
        },
        undefined
    );
}
```

**Назначение**: фабричная функция, создающая экземпляры Document со стандартизированной структурой метаданных.

**Параметры**:
- `pageContent`: очищенное текстовое содержимое
- `source`: URL-адрес или путь к файлу PDF
- `info`: объект с информацией о PDF от pdf-parse
- `metadata`: метаданные PDF (автор, название, дата создания и т. д.)
- `numPages`: общее количество страниц в PDF
- `pageNumber`: номер текущей страницы (необязательный, используется при разделении страниц)

**Структура метаданных**:
```javascript
{
    source: "https://arxiv.org/pdf/2402.19473",  // Origin of the document
    pdf: {
        info: {...},        // PDF properties (version, producer, etc.)
        metadata: {...},    // Author, title, keywords, etc.
        numPages: 22        // Total pages
    },
    loc: {
        pageNumber: 1       // Specific page (for split documents)
    }
}
```

**Преимущества для RAG**: эти метаданные позволяют:
- генерировать ссылки (знать, с какой страницы взята информация)
- фильтровать при поиске (например, «искать только на страницах 5–10»)
- указывать источник в ответах LLM
- отлаживать и отслеживать извлечённое содержимое

### 4. Функция extractTextFromPDF (основная функция)

```javascript
const extractTextFromPDF = async (url, {splitPages} = {splitPages: false}) => {
    // ... implementation
}
```

**Назначение**: основная функция-оркестратор, извлекающая текст из PDF с опциональным разделением по страницам.

#### Параметры

- `url`: строка, содержащая URL-адрес PDF или путь к файлу
- `{splitPages}`: объект параметров с деструктуризированным полем
    - По умолчанию: `{splitPages: false}`
    - При значении `true`: создаются отдельные объекты Document для каждой страницы
    - При значении `false`: создаётся один Document со всем содержимым PDF

#### Разбор реализации

**Шаг 1: Инициализация парсера**
```javascript
const parser = new PDFParse({url});
const info = await parser.getInfo();
const pages = info.total
let doc = []
```
- Создаётся экземпляр PDFParse с URL
- Получается информация о PDF (количество страниц, метаданные)
- Инициализируется пустой массив для хранения объектов Document

**Шаг 2A: Режим разделения страниц** (`splitPages: true`)
```javascript
if (splitPages) {
    for (let i = 0; i < pages; i++) {
        const rawText = (await parser.getText({partial: [i + 1]})).text;
        const cleanedText = cleanText(rawText);

        doc.push(constructDocument(
            cleanedText,
            url,
            info,
            info.metadata,
            pages,
            i + 1  // Page number
        ))
    }
}
```

**Процесс**:
1. Перебираются все страницы (индексация с 0 в цикле, нумерация PDF с 1)
2. Извлекается текст одной страницы с помощью `partial: [i + 1]`
3. Извлечённый текст очищается
4. Для каждой страницы создаётся отдельный объект Document с номером страницы в метаданных
5. Объект добавляется в массив документов

**Применение**: оптимально подходит для:
- больших PDF, где нужен детальный поиск
- точности цитирования (точные ссылки на страницы)
- лучшего семантического разбиения на блоки (с учётом границ страниц)
- эффективности памяти (обработка построчно)

**Шаг 2B: Режим полного документа** (`splitPages: false`)
```javascript
else {
    const rawText = (await parser.getText()).text;
    const cleanedText = cleanText(rawText);

    doc.push(constructDocument(
        cleanedText,
        url,
        info,
        info.metadata,
        pages,
        // No pageNumber parameter
    ))
}
```

**Процесс**:
1. Извлекается весь текст из PDF за один вызов
2. Полный текст очищается
3. Создаётся один объект Document со всем содержимым PDF
4. В метаданных нет `pageNumber`

**Применение**: подходит для:
- небольших PDF
- случаев, когда границы страниц семантически не важны
- более быстрой обработки (однократное извлечение)
- упрощённой логики поиска

**Шаг 3: Возврат документов**
```javascript
return doc;
```
Возвращает массив объектов Document (1 документ или N документов в зависимости от режима)

## Пример использования

```javascript
const url = "https://arxiv.org/pdf/2402.19473"
extractTextFromPDF(url, {splitPages: true}).then(text => {
    console.log(`Extracted ${text.length} document(s). First page snippet:\n`);
    console.log(text[0]?.pageContent.slice(0, 300) + "...");
});
```

**Разбор**:
1. **Выбор URL**: указывает на исследовательскую статью на arXiv о RAG (meta!)
2. **Конфигурация**: используется `splitPages: true` для создания документов на уровне страниц
3. **Обработка промиса**: используется `.then()` для обработки асинхронного результата
4. **Логирование**:
    - выводится общее количество документов (число страниц)
    - отображаются первые 300 символов первой страницы
    - используется безопасный доступ через optional chaining (`?.`)

**Ожидаемый вывод**:
```
Extracted 22 document(s). First page snippet:

Retrieval-Augmented Generation for
AI-Generated Content: A Survey
[Abstract content...]...
```

## Интеграция с конвейером RAG

Этот код представляет собой **первый этап** типичного конвейера RAG:

```
1. Загрузка документов (этот код) ← ВЫ НАХОДИТЕСЬ ЗДЕСЬ
   ↓
2. Разбиение текста на блоки
   ↓
3. Генерация эмбеддингов
   ↓
4. Индексация в векторном хранилище
   ↓
5. Поиск (retrieval)
   ↓
6. Генерация LLM
```

**Следующие шаги** (типичный рабочий процесс):
1. Взять объекты Document из этого кода
2. Разбить большие страницы на более мелкие блоки (например, по 500 токенов)
3. Сгенерировать эмбеддинги для каждого блока с помощью модели эмбеддингов
4. Сохранить эмбеддинги в векторной базе данных (Pinecone, Chroma, FAISS и т. д.)
5. В момент запроса: получить эмбеддинг вопроса пользователя → извлечь релевантные блоки → отправить в LLM

## Паттерны проектирования

### 1. Паттерн «Фабрика»
Функция `constructDocument` выступает в роли фабрики, инкапсулируя логику создания Document и обеспечивая единообразную структуру метаданных.

### 2. Паттерн «Стратегия»
Параметр `splitPages` реализует различные стратегии обработки (постраничная и полнодокументальная) без изменения интерфейса.

### 3. Паттерн Async/Await
Корректно обрабатывает асинхронные операции разбора PDF, делая код более читаемым по сравнению с подходами на основе колбэков.

## Возможные улучшения

1. **Обработка ошибок**: добавить блоки try-catch для сетевых ошибок, недопустимых PDF или сбоев разбора
2. **Отслеживание прогресса**: добавить колбэки для длинных PDF для отслеживания хода извлечения
3. **Стратегия разбиения на блоки**: реализовать семантическое разбиение с учётом границ предложений
4. **Обогащение метаданных**: извлекать название, авторов и аннотацию из метаданных PDF
5. **Кэширование**: кэшировать извлечённый текст для избежания повторной обработки одних и тех же PDF
6. **Пакетная обработка**: поддержка нескольких URL в одном вызове
7. **Генерация ID**: реализовать корректную генерацию идентификаторов документов (например, UUID или хеш на основе содержимого)

## Соображения по производительности

- **Память**: режим разделения страниц более экономичен по памяти для больших PDF
- **Сеть**: загружает весь PDF перед обработкой; для очень больших файлов стоит рассмотреть потоковую передачу
- **Скорость разбора**: pdf-parse достаточно быстр, но может работать медленно для сложных PDF с изображениями
- **Параллелизм**: можно обрабатывать несколько PDF параллельно с помощью `Promise.all()`

## Частые проблемы и решения

### Проблема: искажённые специальные символы
**Решение**: pdf-parse обрабатывает большинство кодировок, но некоторые PDF могут потребовать указания кодировки

### Проблема: таблицы и столбцы перемешаны
**Решение**: извлечение текста из PDF является линейным; стоит рассмотреть специализированные библиотеки для извлечения таблиц

### Проблема: пустые страницы
**Решение**: добавить проверку длины перед созданием объектов Document

### Проблема: ошибки памяти при работе с большими PDF
**Решение**: использовать `splitPages: true` и обрабатывать страницы по отдельности или пакетами
