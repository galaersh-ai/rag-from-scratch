# Генерация эмбеддингов — Концептуальный обзор

## Проблема

В 01_text_similarity_basics мы создали эмбеддинги для 10 документов в памяти. Но что происходит, когда у вас:
- 1 000 документов?
- 10 000 документов?
- Документы, которые меняются со временем?

**Проблемы:**
- ❌ Повторное создание эмбеддингов для всего каждый раз — медленно и расточительно
- ❌ Хранение эмбеддингов в памяти не масштабируется
- ❌ Добавление новых документов требует повторной обработки всего

**Решение:** Персистентность + инкрементальные обновления

---

## Основные концепции

### 1. Пакетная обработка

**Проблема:** Создание эмбеддинга для одного документа за раз неэффективно.

**Решение:** Обработка документов пакетами с отслеживанием прогресса.

```
Before (slow):
for doc in docs:
    embed(doc)      ← Wait for each one
    
After (efficient):
for doc in docs:
    embed(doc)      ← Process sequentially but track progress
    show_progress() ← User knows what's happening
```

**Преимущества:**
- Пользователь видит прогресс (а не замороженный экран)
- Впоследствии можно оптимизировать с помощью параллельной обработки
- Легко приостановить/возобновить

---

### 2. Персистентность (кэширование)

**Золотое правило:** Никогда не пересчитывайте эмбеддинги, если этого не требуется.

**Почему эмбеддинги следует кэшировать:**

| Операция | Время |
|-----------|------|
| Генерация 100 эмбеддингов | 0,5 с |
| Сохранение на диск | 5 мс |
| Загрузка с диска | 2 мс |

**Загрузка в 250 раз быстрее генерации!**

**Ментальная модель:**
```
First Run:
  Generate embeddings (slow) → Save to disk

Every Other Run:
  Load from disk (fast) → Use immediately
```

**Практический эффект:**
- Разработка: Перезапуск приложения 100 раз → экономия 50 с каждый раз = 83 минуты сэкономлено
- Продакшен: Перезапуск API → Пользователи не ждут повторного создания эмбеддингов

---

### 3. Инкрементальные обновления

**Сценарий:** У вас есть 1 000 документов с эмбеддингами. Вы добавляете 10 новых.

**Плохой подход:**
```
Re-embed all 1,010 documents
Time: ~5 seconds
```

**Хороший подход:**
```
1. Load existing 1,000 embeddings (2ms)
2. Check which documents are new (1ms)
3. Embed only 10 new documents (50ms)
4. Merge and save (5ms)
Time: ~58ms (86x faster!)
```

**Как это работает:**
1. Каждый документ получает уникальный ID
2. Эмбеддинги хранятся вместе со своими идентификаторами
3. При добавлении документов проверяются ID на соответствие существующим
4. Обрабатываются только отсутствующие

**Примеры использования:**
- Ежедневное обновление документов (новости, отчёты)
- Контент, загружаемый пользователями
- Растущие базы знаний

---

### 4. Форматы хранения

**Формат JSON:**
```json
{
  "version": "1.0",
  "model": "bge-small-en-v1.5",
  "dimensions": 384,
  "embeddings": [
    {
      "id": "doc_0",
      "content": "Text here...",
      "embedding": [0.023, -0.156, ...]
    }
  ]
}
```

**Плюсы:**
- ✅ Читаемость для человека
- ✅ Легкость отладки
- ✅ Работает везде
- ✅ Можно проверить в текстовом редакторе

**Минусы:**
- ❌ Большой размер файла (~11 КБ на документ)
- ❌ Медленный парсинг (накладные расходы на разбор JSON)

**Когда использовать:**
- Разработка и отладка
- Небольшие наборы данных (< 1 000 документов)
- Необходимость ручной проверки эмбеддингов
- Обмен данными между различными системами

---

### 5. Управление метаданными

**Почему метаданные важны:**

Одних эмбеддингов недостаточно. Необходимо знать:
- **Какой** документ представляет данный эмбеддинг
- **Откуда** он был получен (источник, номер страницы)
- **Когда** он был создан
- **Как** он был создан (модель, размер фрагмента)

**Пример:**
```javascript
{
  id: "doc_42",
  embedding: [...],
  metadata: {
    content: "Machine learning is...",
    source: "ml_guide.pdf",
    page: 5,
    chunk: 2,
    created: "2024-11-04T10:30:00Z",
    model: "bge-small-en-v1.5"
  }
}
```

**Почему каждое поле важно:**
- `id`: Дедупликация и обновления
- `content`: Отображение в результатах поиска
- `source`: Указание источника информации
- `page/chunk`: Навигация к точному расположению
- `created`: Определение времени для повторной генерации
- `model`: Обеспечение совместимости

## Связь с RAG

### Как это вписывается в RAG

```
┌─────────────────────────────────────────────────────────┐
│              RAG PIPELINE                               │
└─────────────────────────────────────────────────────────┘

INDEXING (One-time or periodic)
├─ Load documents
├─ Split into chunks
├─ Generate embeddings  ← THIS EXAMPLE
├─ Save embeddings      ← THIS EXAMPLE
└─ Store in vector DB

RETRIEVAL (Per query)
├─ User asks question
├─ Embed question
├─ Search vector DB
└─ Return relevant chunks

GENERATION (Per query)
├─ Combine query + chunks
└─ LLM generates answer
```

**Данный пример сосредоточен на:**
- Эффективной генерации эмбеддингов
- Персистентном хранении
- Управлении обновлениями

## Соображения масштабирования

### Малый масштаб (< 1 000 документов)

**Стратегия:** Простое файловое хранилище

```javascript
// Generate once
embeddings = await generateEmbeddings(context, documents);
await saveEmbeddingsJSON(embeddings, 'embeddings.json');

// Load many times
embeddings = await loadEmbeddingsJSON('embeddings.json');
```

**Подходит для:**
- Персональных проектов
- Небольших корпоративных баз знаний
- Прототипов и MVP

---

### Средний масштаб (1 000 – 100 000 документов)

**Стратегия:** Инкрементальные обновления + раздельное хранение

```javascript
// Split into multiple files
await saveEmbeddingsJSON(batch1, 'embeddings_1.json');
await saveEmbeddingsJSON(batch2, 'embeddings_2.json');

// Incremental updates
newEmbeddings = await incrementalEmbedding(context, newDocs, 'embeddings.json');
```

**Стоит учесть:**
- Несколько файлов хранилища
- Сжатие
- Фоновая обработка
- Инкрементальные обновления

---

### Большой масштаб (> 100 000 документов)

**Стратегия:** Переход на векторную базу данных

**Почему нужна векторная база данных:**
- ✅ Оптимизированный поиск по сходству (алгоритмы HNSW, IVF)
- ✅ Обработка миллионов векторов
- ✅ Встроенная персистентность и резервное копирование
- ✅ Распределённая архитектура
- ✅ Обновления в реальном времени

**Популярные варианты:**
- LanceDB (встроенная, без сервера)
- Qdrant (самостоятельный хостинг, высокая производительность)
- Chroma (удобна для разработчиков)
- Milvus (корпоративный масштаб)

## Шаблоны оптимизации производительности

### Шаблон 1: Ленивая загрузка

```javascript
let embeddingsCache = null;

async function getEmbeddings() {
    if (!embeddingsCache) {
        embeddingsCache = await loadEmbeddingsJSON('embeddings.json');
    }
    return embeddingsCache;
}
```

**Преимущества:**
- Загрузка только при необходимости
- Загрузка один раз, многократное использование
- Снижение потребления памяти

---

### Шаблон 2: Потоковые обновления

```javascript
async function* generateEmbeddingsStream(context, documents) {
    for (const doc of documents) {
        const embedding = await context.getEmbeddingFor(doc.pageContent);
        yield { doc, embedding };
    }
}

// Use it
for await (const {doc, embedding} of generateEmbeddingsStream(context, docs)) {
    await saveToDatabase(embedding);
    console.log(`Processed ${doc.metadata.id}`);
}
```

**Преимущества:**
- Обработка больших наборов данных
- Не нужно удерживать всё в памяти
- Возможность сохранения постепенно

---

### Шаблон 3: Версионирование

```javascript
const embeddingData = {
    version: '2.0',
    model: 'bge-small-en-v1.5',
    created: new Date().toISOString(),
    embeddings: [...]
};

// Load with version check
const data = await loadEmbeddingsJSON('embeddings.json');
if (data.version !== '2.0') {
    console.log('Old version detected, regenerating...');
    await regenerateEmbeddings();
}
```

**Преимущества:**
- Обнаружение несовместимых эмбеддингов
- Определение времени для повторной генерации
- Отслеживание изменений схемы

## Ключевые выводы

### 1. Всегда кэшируйте эмбеддинги
**Почему:** В 100–1000 раз быстрее повторной генерации

### 2. Используйте инкрементальные обновления
**Почему:** Обрабатывайте только новое

### 3. Включайте подробные метаданные
**Почему:** Делает результаты полезными (а не просто числами)

### 4. Отслеживайте версии
**Почему:** Определяйте, когда необходима повторная генерация

### 5. Выбирайте хранилище в зависимости от масштаба
- Малый: файлы JSON
- Средний: сжатые файлы + инкрементальные обновления
- Большой: векторные базы данных
