# Мультизапросный ретривер: общая картина

Это руководство объясняет *зачем* и *как* работает мультизапросный ретривер (multi-query retrieval) с помощью интуитивных концепций и аналогий из реальной жизни. Идеально подходит для понимания идей перед погружением в код.

---

## Основная идея

Представьте, что вы ищете информацию в библиотеке. Вы могли бы:

1. **Одиночный запрос**: подойти к библиотекарю и сказать «Мне нужны книги о веб-разработке»
    - Библиотекарь даёт 3–5 книг
    - Они могут все быть посвящены одному аспекту (например, только JavaScript)

2. **Мультизапрос**: задать несколько конкретных вопросов:
    - «Какие книги охватывают веб-архитектуру?»
    - «А что насчёт масштабирования приложений?»
    - «И практики тестирования?»
    - Теперь вы получаете 10–15 книг, охватывающих все аспекты!

**Мультизапросный ретривер — это задание нескольких сфокусированных вопросов вместо одного широкого.**

---

## Почему это важно?

### Проблема одиночного запроса

Когда вы ищете одним запросом, вы сталкиваетесь с тремя основными проблемами:

**1. Туннельное зрение**
```
Query: «How do I cook a healthy meal?»
Result: Может вернуть только рецепты, пропустив руководства по питанию или планированию meals
```

**2. Разрыв в терминологии**
```
Ваши слова: «make apps faster»
Слова в документе: «optimize performance», «improve efficiency»
Результат: несовпадение означает, что вы пропускаете релевантный контент
```

**3. Перегрузка сложностью**
```
Query: «Compare React vs Vue for building scalable e-commerce sites with good SEO»
Результат: поиск путается от слишком многих концепций одновременно
```

### Решение — мультизапрос

**Вместо одного сложного вопроса задайте несколько простых:**

```
Одиночный (избегать): «Compare React vs Vue for building scalable e-commerce sites with good SEO»
Мультизапрос (предпочтительно):
   1. «What is React?»
   2. «What is Vue?»
   3. «React vs Vue comparison»
   4. «Building scalable web applications»
   5. «E-commerce platform architecture»
   6. «SEO best practices for web apps»
```

Каждый сфокусированный вопрос извлекает разные релевантные документы. Вместе они обеспечивают всестороннее покрытие.

## Ключевые концепции

### 1. Декомпозиция запроса (query decomposition)

**Идея:** разбейте сложные вопросы на более простые части.

**Аналогия из реальной жизни:**

Представьте планирование поездки:
- Плохо: «How do I plan a trip to Europe?»
- Хорошо:
    - «What are the best cities to visit in Europe?»
    - «How to book affordable flights?»
    - «What travel documents do I need?»
    - «Where should I stay in each city?»

Каждый вопрос прощеanswered, и вместе они охватывают всё, что нужно.

**В RAG:**
```
Сложный: «How do I build a secure, scalable web app with good performance?»

Декомпозированный:
1. «Web application architecture patterns»
2. «Security best practices for web apps»
3. «Application scalability techniques»
4. «Web performance optimization»
```

**Результат:** вместо 3 смазанно релевантных документов вы получаете 2–3 Highly релевантных документа на под-запрос = 8–12 релевантных документов всего!

---

### 2. Расширение запроса (query expansion)

**Идея:** скажите одно и то же разными способами.

**Аналогия из реальной жизни:**

Если вы ищете ключи от машины, вы можете спросить:
- «Where are my keys?»
- «Have you seen my car keys?»
- «Did I leave my keys here?»
- «Anyone know where my keys are?»

Разные люди могут отреагировать на разные формулировки.

**В RAG:**
```
Оригинал: «How to optimize database queries?»

Расширенный:
1. «How to optimize database queries?»
2. «Database performance improvement techniques»
3. «Speed up slow database operations»
4. «Query optimization best practices»
```

**Почему это работает:**
- разные документы используют разную терминологию
- «optimize» vs «improve» vs «speed up» vs «enhance»
- каждая формулировка ловит разные документы
- комбинация = всесторонние результаты

**Визуальное представление:**
```
Query 1: «optimize database» → Находит: Doc A, Doc C, Doc E
Query 2: «improve performance» → Находит: Doc B, Doc C, Doc F
Query 3: «speed up queries» → Находит: Doc A, Doc D, Doc G

Комбинированные уникальные результаты: Doc A, B, C, D, E, F, G
```

---

### 3. Параллельное выполнение (parallel execution)

**Идея:** делайте несколько вещей одновременно вместо поочерёдно.

**Аналогия из реальной жизни:**

Представьте, что вы повар, готовящий обед:

**Последовательно (медленно):**
```
1. Кипятить воду → ждать 10 минут
2. Резать овощи → ждать 5 минут
3. Готовить мясо → ждать 15 минут
Итого: 30 минут
```

**Параллельно (быстро):**
```
1. Начать кипятить воду И резать овощи И готовить мясо
   Всё одновременно!
Итого: 15 минут (время самой длинной задачи)
```

**В RAG:**
```javascript
// Последовательно: 400ms всего
Query 1 → ждать 100ms
Query 2 → ждать 100ms
Query 3 → ждать 100ms
Query 4 → ждать 100ms

// Параллельно: 100ms всего
Все запросы стартуют вместе → ждать 100ms (для самого медленного)
```

**Ускорение:** в 3–4 раза быстрее!

---

### 4. Фузия результатов (result fusion)

**Идея:** интеллектуально объедините несколько списков результатов в один ранжированный список.

**Аналогия из реальной жизни:**

Вы выбираете ресторан и спрашиваете трёх друзей о рекомендациях:

**Список друга 1:**
1. Итальянское место (обожает)
2. Суши-бар (нравится)
3. Бургерная (нормально)

**Список друга 2:**
1. Суши-бар (обожает)
2. Тайский ресторан (нравится)
3. Итальянское место (нормально)

**Список друга 3:**
1. Итальянское место (обожает)
2. Бургерная (нравится)
3. Мексиканское место (нормально)

**Как их объединить?**

**Метод 1: усреднение оценок** (проблематично)
- Проблема: разные друзья используют разные шкалы оценок
- Друг 1 щедр (ставит 8–10)
- Друг 2 строг (ставит 4–6)
- Усреднение несправедливо!

**Метод 2: на основе ранжирования (RRF)** (рекомендуется)
- Итальянское место: встречается в 3 списках (1-е, 3-е, 1-е) → стабильно хорошее!
- Суши-бар: встречается в 2 списках (2-е, 1-е) → тоже отличное
- Бургерная: встречается в 2 списках (3-е, 2-е) → нормально

**Итоговое ранжирование:** Итальянское место > Суши-бар > Бургерная

**В RAG:**

Именно это делает консенсусный рейтинг (Reciprocal Rank Fusion, RRF):
- смотрит на ранжирования, а не на оценки
- документы, встречающиеся в нескольких списках результатов, получают более высокий рейтинг
- устойчив к различиям в шкалах оценок

---

### 5. Консенсусный рейтинг (RRF) — подробный разбор

**Формула:** `Score = 1 / (k + rank + 1)`

**Почему эта формула работает:**

Представьте это как «бонус за последовательность»:

```
Документ появляется один раз на 1-м месте:
Score = 1/(60+0+1) = 0.016

Документ появляется три раза (места 0, 1, 2):
Score = 1/61 + 1/62 + 1/63 = 0.049

Чем чаще он появляется, тем выше общий счёт!
```

**Визуальный пример:**
```
Результаты Query 1:          Результаты Query 2:          Результаты Query 3:
1. doc_A                  1. doc_B                  1. doc_A
2. doc_B                  2. doc_A                  2. doc_C
3. doc_C                  3. doc_D                  3. doc_B

Расчёт RRF:
doc_A: 1/61 + 1/62 + 1/61 = 0.049 (появляется 3 раза) - Лучший
doc_B: 1/62 + 1/61 + 1/63 = 0.048 (появляется 3 раза)
doc_C: 1/63 + 1/63 = 0.032 (появляется 2 раза)
doc_D: 1/63 = 0.016 (появляется 1 раз)
```

**Ключевой инсайт:** doc_A стабильно занимает высокие места, поэтому побеждает!

### 6. Ретривер на основе перспектив (perspective-based retrieval)

**Идея:** рассмотрите одну тему с разных сторон.

**Аналогия из реальной жизни:**

Вы рассматриваете покупку нового автомобиля. Вы спрашиваете:
- **Продавца:** «Почему мне стоит купить эту машину?» → слышите преимущества
- **Механика:** «Какие проблемы у этой машины?» → слышите недостатки
- **Текущего владельца:** «Как она в повседневной эксплуатации?» → слышите реальность
- **Автожурналиста:** «Как она сравнивается с конкурентами?» → слышите контекст

Каждая перспектива даёт вам разную ценную информацию!

**В RAG:**
```
Тема: «microservices architecture»

Разные перспективы:
1. Преимущества: «What are the advantages of microservices?»
2. Проблемы: «What are common microservices problems?»
3. Реализация: «How to implement microservices?»
4. Сравнение: «Microservices vs monolithic architecture?»
```

**Результат:** сбалансированное, всестороннее понимание вместо односторонней информации.

---

### 7. Взвешенная фузия запросов (weighted query fusion)

**Идея:** не все запросы одинаково важны.

**Аналогия из реальной жизни:**

Вы планируете вечеринку и собираете мнения:
- ваши предпочтения: 100% важны (это ВАША вечеринка!)
- предложения лучшего друга: 80% важны (они вас хорошо знают)
- идеи знакомого: 30% важны (приятно рассмотреть)

Вы взвешиваете вклад каждого человека соответствующим образом.

**В RAG:**
```
Основной запрос: «React state management»                 (вес: 1.0)
Связанный запрос: «React hooks for state»               (вес: 0.8)
Контекстный запрос: «JavaScript state patterns»           (вес: 0.5)
Фоновый запрос: «Frontend architecture basics»     (вес: 0.3)
```

**Подсчёт:**
```
Документ найден в:
- Основной запрос (оценка 0.95)      → вклад: 0.95 × 1.0 = 0.95
- Связанный запрос (оценка 0.88)   → вклад: 0.88 × 0.8 = 0.70
Итоговая взвешенная оценка: 1.65
```

Это гарантирует, что основной запрос доминирует, но связанные запросы добавляют контекст.

---

### 8. Адаптивный выбор стратегии (adaptive strategy selection)

**Идея:** разные вопросы требуют разных подходов.

**Аналогия из реальной жизни:**

Разные проблемы требуют разных инструментов:
- забить гвоздь → молоток
- затянуть винт → отвёртка
- распилить дерево → пила

Вы не будете использовать молоток для всего!

**В RAG:**
```
Простой запрос: «What is React?»
→ Стратегия: одиночный запрос (быстро, просто)

Умеренный запрос: «How to optimize React performance?»
→ Стратегия: расширение запроса (лучшее покрытие)

Сложный запрос: «What's the difference between REST and GraphQL?»
→ Стратегия: на основе перспектив (сбалансированное сравнение)

Очень сложный: «How do I build scalable apps with microservices, CI/CD, and testing?»
→ Стратегия: многочастичная декомпозиция (всесторонняя)
```

**Автоматическое определение:**
```javascript
if (wordCount < 5) → Простой → Одиночный запрос
else if (содержит «difference» или «vs») → Сравнение → На основе перспектив
else if (содержит «and» + длинный) → Многочастичный → Декомпозиция
else → Умеренный → Расширение запроса
```

## Собираем всё вместе: реальный пример

Пройдём через полный конвейер мультизапросного ретривера:

### Запрос пользователя
```
«How do I build a scalable web application with good testing practices?»
```

### Шаг 1: Анализ запроса
```
- 11 слов (умеренная сложность)
- Содержит «and» (несколько тем)
- Широкий охват
→ Решение: использовать декомпозицию запроса
```

### Шаг 2: Декомпозиция
```
Под-запросы:
1. «Web application architecture patterns»
2. «Building scalable applications»
3. «Software testing best practices»
4. «Automated testing implementation»
```

### Шаг 3: Выполнение запросов (параллельно)
```
Запускаем все 4 одновременно:
├─ Query 1 → находит 3 документа об архитектуре
├─ Query 2 → находит 3 документа о масштабируемости
├─ Query 3 → находит 3 документа о тестировании
└─ Query 4 → находит 3 документа об автоматизации

Итого: 12 документов извлечено
Время: 120ms (vs 400ms последовательно)
```

### Шаг 4: Дедупликация
```
12 документов → некоторое перекрытие

Дедупликация (оставляем наивысшую оценку на документ):
12 документов → 8 уникальных документов
```

### Шаг 5: Применение RRF-фузии
```
Расчёт RRF-оценок:

doc_010 (microservices): встречается в запросах 1, 2 → оценка: 0.032
doc_016 (unit testing): встречается в запросах 3, 4 → оценка: 0.032
doc_022 (kubernetes): встречается в запросе 2 → оценка: 0.016
doc_017 (integration): встречается в запросах 3, 4 → оценка: 0.031
...
```

### Шаг 6: Возврат ранжированных результатов
```
1. [0.032] Архитектура микросервисов (охватывает масштабируемость)
2. [0.032] Практики unit-тестирования (охватывает тестирование)
3. [0.031] Интеграционное тестирование (охватывает тестирование)
4. [0.028] Docker-контейнеры (охватывает архитектуру)
5. [0.025] CI/CD-конвейеры (охватывает автоматизацию)
```

### Качество результатов
```
- Всестороннее покрытие (архитектура + масштабируемость + тестирование)
- Разнообразные перспективы (несколько аспектов каждой темы)
- Высокая релевантность (фокусированные под-запросы)
- Быстрое выполнение (параллельный ретривер)
```

## Когда использовать мультизапросный ретривер

### Хорошие случаи применения

**1. Сложные вопросы**
```
«How do I migrate from monolithic to microservices while maintaining uptime?»
→ Несколько концепций, нужна декомпозиция
```

**2. Вопросы для сравнения**
```
«Should I use SQL or NoSQL for my project?»
→ Нужны сбалансированные перспективы
```

**3. Обучение/исследование**
```
«Tell me everything about Kubernetes»
→ Нужно всестороннее покрытие
```

**4. Принятие решений**
```
«What are the pros and cons of serverless architecture?»
→ Нужны множественные точки зрения
```

### Плохие случаи применения

**1. Простые фактические вопросы**
```
«What is the capital of France?»
→ Одиночного запроса достаточно, декомпозиция тратит время
```

**2. Очень специфичные запросы**
```
«What's the error code for HTTP unauthorized?»
→ Нужно точное совпадение, расширение не поможет
```

**3. Приложения с жёсткими ограничениями по времени**
```
Чатботы реального времени, требующие <100ms
→ Одиночный запрос быстрее
```

**4. Ограниченные вычислительные ресурсы**
```
Запуск на мобильном устройстве с ограниченными ресурсами
→ Множественные запросы разряжают батарею/канал
```

## Компромиссы

### Качество vs. скорость
```
Одиночный запрос:
- Скорость: 5/5 (50ms)
- Качество: 3/5 (может пропустить аспекты)
- Сложность: 1/5 (просто)

Мультизапрос:
- Скорость: 3/5 (150–300ms)
- Качество: 5/5 (всесторонне)
- Сложность: 4/5 (сложнее)
```

**Рекомендация:** используйте адаптивную стратегию для автоматического баланса!

### Ширина vs. точность
```
Одиночный запрос:
- Точность (precision): 4/5 (фокусированный)
- Полнота (recall): 3/5 (может пропустить документы)
- Разнообразие: 2/5 (ограниченные перспективы)

Мультизапрос:
- Точность: 3/5 (некоторые нерелевантные)
- Полнота: 5/5 (ловит больше)
- Разнообразие: 5/5 (множественные перспективы)
```

**Рекомендация:** используйте для сложных запросов, где важна полнота!

## Ключевые выводы

### Главные идеи

1. **Одного вопроса недостаточно** — сложные запросы нуждаются в декомпозиции
2. **Разные слова, одно значение** — расширение запроса обрабатывает разрывы в терминологии
3. **Делайте вещи одновременно** — параллельное выполнение ускоряет работу
4. **Ранжирования важнее оценок** — RRF комбинирует результаты устойчиво
5. **Разные вопросы требуют разных стратегий** — адаптируйте подход к сложности
6. **Множественные перспективы = полная картина** — видите всю тему, а не только один угол

### Ментальная модель

Думайте о мультизапросном ретривере как:

**Расследование истории журналистом:**
- не спрашивайте одного человека об одном
- спрашивайте нескольких людей (перспективы)
- спрашивайте одно и то же разными способами (расширение)
- задавайте уточняющие вопросы (декомпозиция)
- говорите со всеми сразу (параллельно)
- взвешивайте источники по достоверности (взвешенная фузия)
- ищите согласованную информацию (RRF)

**Результат:** всесторонняя, точная, сбалансированная история!

### Когда использовать

```
Используйте мультизапрос, когда:
- Запрос сложный (несколько концепций)
- Нужно всестороннее покрытие
- Сравнение или принятие решений
- Сценарии обучения/исследования
- Одиночный запрос даёт плохие результаты

Пропускайте мультизапрос, когда:
- Запрос простой (фактический поиск)
- Нужен ультрабыстрый ответ (<100ms)
- Ограниченные вычислительные ресурсы
- Одиночного запроса достаточно
```

## Практические советы по реализации

### Начните просто
```
1. Начните с одиночного запроса
2. Если результаты плохие, добавьте расширение запроса (2–3 вариации)
3. Если всё ещё плохо, попробуйте декомпозицию
4. Используйте RRF для комбинирования результатов
5. Измеряйте и итерируйте!
```

### Оптимизируйте постепенно
```
Фаза 1: заставьте работать (последовательное выполнение)
Фаза 2: сделайте быстрым (параллельное выполнение)
Фаза 3: сделайте умным (адаптивная стратегия)
Фаза 4: сделайте эффективным (кэширование, ограничения)
```

### Отслеживайте и адаптируйтесь
```
Отслеживайте метрики:
├─ Среднее время запроса
├─ Количество возвращённых результатов
├─ Удовлетворённость пользователей (click-through rate)
└─ Наиболее частые паттерны запросов

Корректируйте на основе данных:
├─ Если слишком медленно → уменьшите количество под-запросов
```

## Часто задаваемые вопросы

**В: Разве это не просто усложнение поиска?**
О: Для простых запросов — да! Используйте одиночный запрос. Для сложных запросов мультизапросный ретривер значительно улучшает результаты. Используйте адаптивно.

**В: Сколько под-запросов использовать?**
О: Оптимально 3–5 под-запросов. Более 7 обычно не помогает и замедляет работу.

**В: Что делать, если под-запросы перекрываются?**
О: Это нормально! Дедупликация это обрабатывает. Перекрывающиеся запросы даже помогают — они усиливают важные документы.

**В: Параллельное выполнение всегда быстрее?**
О: Почти всегда да. Единственное исключение — если у вас очень ограниченные ресурсы или бэкенд не поддерживает параллельные запросы.

**В: Всегда ли использовать RRF для фузии?**
О: RAF — отличный вариант по умолчанию. Используйте взвешенную фузию, когда хотите приоритизировать одни запросы над другими.

**В: Как узнать, работает ли это?**
О: Сравните результаты одиночного и мультизапроса:
- стали ли результаты мультизапроса более всесторонними?
- находят ли пользователи то, что им нужно, быстрее?
- приемлема ли задержка?

Если да на все три — это работает!
