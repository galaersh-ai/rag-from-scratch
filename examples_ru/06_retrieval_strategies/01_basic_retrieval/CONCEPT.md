# Базовая стратегия извлечения — концептуальный обзор

## Что такое базовое извлечение в RAG?

**Базовое извлечение (Basic Retrieval)** — это фундаментальная стратегия для систем Retrieval-Augmented Generation (RAG). Она объединяет два ключевых компонента:

1. **Извлечение (Retrieval)**: поиск релевантных документов из базы знаний с помощью поиска по векторному сходству
2. **Усиленная генерация (Augmented Generation)**: использование LLM для генерации ответов на основе извлечённого контекста

Это самый простой и распространённый паттерн RAG, который лежит в основе всех продвинутых стратегий извлечения.

## Конвейер RAG

### Трёхэтапный процесс

```
┌─────────────────┐
│  1. RETRIEVE    │  Поиск релевантных документов по векторному сходству
│                 │
│  User Query     │
│       ↓         │
│  Embed Query    │
│       ↓         │
│  Vector Search  │
│       ↓         │
│  Top-k Results  │
└─────────────────┘
         ↓
┌─────────────────┐
│  2. ASSEMBLE    │  Формирование контекста из извлечённых документов
│                 │
│  Retrieved Docs │
│       ↓         │
│  Format Context │
│       ↓         │
│  Context String │
└─────────────────┘
         ↓
┌─────────────────┐
│  3. GENERATE    │  LLM генерирует ответ с использованием контекста
│                 │
│  Query + Context│
│       ↓         │
│  LLM Inference  │
│       ↓         │
│  Final Answer   │
└─────────────────┘
```

### Почему это работает

**Без RAG (только LLM):**
```javascript
Question: "What is our company's refund policy?"
LLM: "I don't have specific information about your company..."
```
❌ Обобщённый ответ, без конкретных знаний

**С RAG (извлечение + LLM):**
```javascript
Question: "What is our company's refund policy?"
Retrieved: [Your company's actual refund policy document]
LLM: "According to your policy, refunds are available within 30 days..."
```
✅ Конкретный, точный ответ, основанный на ваших данных

## Основные компоненты

### 1. Обработка запроса

**Что происходит:**
- Пользователь отправляет вопрос на естественном языке
- Вопрос преобразуется в вектор (эмбеддинг) (той же моделью, что и для документов)
- Вектор представляет семантическое значение вопроса

**Пример:**
```javascript
Query: "How do I reset my password?"
↓
Embedding: [0.234, -0.891, 0.432, ..., 0.123] // 384 dimensions
```

**Ключевая особенность:** Одна и та же модель эмбеддингов должна использоваться как для документов, так и для запросов.

### 2. Поиск по векторному сходству

**Что происходит:**
- Вектор запроса сравнивается со всеми векторами документов в базе
- Документы ранжируются по оценке сходства (косинусное сходство)
- Извлекаются k наиболее релевантных документов

**Оценки сходства:**
```
Диапазон оценок   Интерпретация
────────────────────────────
0.8 - 1.0      Высокая релевантность
0.6 - 0.8      Релевантность  
0.4 - 0.6      Умеренная релевантность
0.2 - 0.4      Слабая релевантность
0.0 - 0.2      Не релевантно
```

**Пример:**
```javascript
Query: "machine learning algorithms"

Results:
1. [0.87] "Machine learning uses algorithms to learn from data..."
2. [0.76] "Neural networks are a type of ML algorithm..."
3. [0.68] "Supervised learning trains models on labeled data..."
4. [0.42] "Python is popular for data science..."
5. [0.21] "Databases store structured information..."
```

Первые 3 документа релевантны; 4-5 — шум.

### 3. Параметр k

**k** определяет количество документов для извлечения.

**Компромиссы:**

| Значение k | Плюсы | Минусы | Применение |
|---------|------|------|----------|
| k=1 | Быстро, сфокусировано | Может не хватить контекста | Быстрые запросы |
| k=3 | Баланс | Хороший параметр по умолчанию | Универсальный вариант |
| k=5 | Больший охват | Некоторый шум | Сложные запросы |
| k=10+ | Максимальный контекст | Много шума | Редко; для переупорядочивания |

**Рекомендации:**
- **Начните с k=3** в качестве параметра по умолчанию
- **Увеличьте k**, если запросы требуют нескольких точек зрения
- **Извлекайте больше** (например, k=10), а затем фильтруйте по порогу сходства
- **Учитывайте размер окна контекста**: больше документов = больше токенов

### 4. Формирование контекста

**Что происходит:**
- Извлечённые документы форматируются в единую строку контекста
- Контекст включает содержимое документов (и опционально метаданные)
- Контекст вставляется в промпт LLM

**Простой формат:**
```
Context:
[1] Python is a programming language...
[2] JavaScript runs in web browsers...
[3] SQL manages database queries...
```

**Расширенный формат:**
```
Context (3 documents, sorted by relevance):

[Document 1 - Similarity: 0.87]
Title: Introduction to Python
Content: Python is a high-level programming language...
Category: Programming | Date: 2024

[Document 2 - Similarity: 0.76]
Title: JavaScript Basics
Content: JavaScript enables interactive web pages...
Category: Programming | Date: 2024
```

**Ключевые моменты:**
- Порядок важен: наиболее релевантные документы в начале
- Сохраняйте единый формат
- Включайте метаданные, если они полезны для качества ответа
- Контролируйте общее количество токенов

### 5. Инженерия промптов для RAG

**Базовая структура:**
```
System: You are a helpful assistant. Answer based on the context.

Context:
[Retrieved documents here]

Question: [User's question]

Answer:
```

**Лучшие практики:**

✅ **Чёткие инструкции:**
```
"Use the following context to answer the question.
If the context doesn't contain relevant information, say so."
```

✅ **Указание источников (опционально):**
```
"Answer the question and cite which document(s) you used."
```

✅ **Определение границ:**
```
"Only answer based on the provided context. Don't use external knowledge."
```

❌ **Слишком расплывчато:**
```
"Here's some info: [context]. Question: [question]"
```

### 6. Генерация ответа

**Что происходит:**
- LLM получает: системный промпт + контекст + вопрос
- LLM генерирует ответ, основанный на контексте
- Ответ должен быть релевантным, точным и, по возможности, указывать на контекст

**Без контекста:**
```
Q: "What is Quantum Flux Architecture?"
A: "I don't have information about that specific architecture."
```

**С контекстом:**
```
Context: "Quantum Flux Architecture is a novel design pattern..."
Q: "What is Quantum Flux Architecture?"
A: "Quantum Flux Architecture is a novel design pattern that..."
```

## Почему базовое извлечение работает

### 1. Основывает LLM на фактах

**Проблема:** LLM могут галлюцинировать или предоставлять устаревшую информацию.

**Решение:** RAG обеспечивает конкретный, актуальный контекст из вашей базы знаний.

**Пример:**
```
LLM Only: "The capital of Country X is probably City Y."
RAG: "According to our database, the capital of Country X is City Z."
```

### 2. Добавляет специфические знания предметной области

**Проблема:** LLM не обладают знаниями о вашей компании, продуктах или собственных данных.

**Решение:** RAG извлекает релевантные внутренние документы.

**Пример:**
```
LLM Only: "I don't know your company's policies."
RAG: "Based on your employee handbook, the vacation policy is..."
```

### 3. Обеспечивает указание источников

**Проблема:** Трудно проверить утверждения LLM.

**Решение:** RAG показывает, какие документы были использованы.

**Пример:**
```
Answer: "The project deadline is March 15."
Source: [Internal Email from Manager, Feb 20, 2024]
```

### 4. Поддерживает обновления в реальном времени

**Проблема:** LLM имеют дату отсечения обучения.

**Решение:** Базу знаний можно обновить в любой момент.

**Пример:**
```
New document added today → Immediately available in RAG system
```

## Типичные проблемы и решения

### Проблема 1: Отсутствие релевантных документов

**Ситуация:** Запрос выходит за рамки базы знаний.

**Неправильная обработка:**
```javascript
// Returns low-similarity docs anyway
const results = await search(query, k=3);
// LLM tries to answer with irrelevant context → Wrong answer
```

**Правильная обработка:**
```javascript
const results = await search(query, k=5);
const filtered = results.filter(r => r.similarity > 0.4);

if (filtered.length === 0) {
    return "I don't have information about that in my knowledge base.";
}
```

**Лучшая практика:** Установите порог сходства (0,3–0,5).

### Проблема 2: Слишком большой контекст

**Ситуация:** Слишком много документов превышают размер окна контекста LLM.

**Проблема:**
```javascript
k=10, each doc = 500 tokens
Total context = 5000 tokens
Question + System = 100 tokens
Answer space = 0 tokens left! ❌
```

**Решение:**
```javascript
k=3, each doc = 500 tokens
Total context = 1500 tokens
Question + System = 100 tokens  
Answer space = 2000+ tokens ✅
```

**Лучшие практики:** 
- Контролируйте использование токенов
- Сохраняйте контекст < 50% от общего размера окна
- Приоритет качества над количеством

### Проблема 3: Избыточная информация

**Ситуация:** Результаты top-k слишком похожи друг на друга.

**Проблема:**
```
1. [0.89] "Python is a programming language..."
2. [0.88] "Python is a high-level programming language..."
3. [0.87] "Python, a programming language, is..."
```
Все говорят об одном и том же → Потеря контекста!

**Решение:** Внедрите разнообразие в продвинутых стратегиях извлечения (следующая глава).

**Базовая смягчающая мера:** Используйте более широкий k, затем отфильтруйте дубликаты.

### Проблема 4: Несоответствие запроса и документа

**Ситуация:** Формулировка вопроса не совпадает с формулировкой в документе.

**Пример:**
```
Query: "How can I get my money back?"
Document: "Refund Policy: Returns accepted within 30 days"
Similarity: 0.45 (Medium, might not be retrieved with k=3)
```

**Решения:**
- Используйте k=5-10 для повышения полноты (recall)
- Расширение запросов (продвинутая техника)
- Улучшени chunking документов
- Улучшение эмбеддингов дообучением

## Характеристики производительности

### Типичная разбивка задержек

```
Total RAG Query: ~500ms
├── Embed Query:        45ms  (9%)
├── Vector Search:       5ms  (1%)
├── Context Assembly:    2ms  (<1%)
└── LLM Generation:    448ms  (90%)
```

**Ключевой момент:** Генерация LLM доминирует по задержке!

### Приоритеты оптимизации

1. **Используйте более быструю/меньшую модель LLM** (наибольшее влияние)
2. **Кэшируйте частые запросы**
3. **Уменьшите k** (минимальное влияние)
4. **Оптимизируйте эмбеддинги** (небольшое влияние)

### Характеристики масштабирования

| Документы | Время поиска | Качество |
|-----------|-------------|---------|
| 100 | 2 мс | Хорошее |
| 1 000 | 5 мс | Хорошее |
| 10 000 | 10 мс | Хорошее |
| 100 000 | 20 мс | Хорошее |
| 1 000 000+ | 30–50 мс | Хорошее |

**Особенность:** Векторный поиск хорошо масштабируется с алгоритмом HNSW.

## Когда базового извлечения достаточно

### ✅ Подходящие варианты использования

**1. Внутренняя база знаний**
- Корпоративные вики
- Документация
- ЧаВо (FAQ)

- Политики и процедуры

**2. Ответы на вопросы**
- Поддержка клиентов
- Онбординг сотрудников
- Информация о продуктах
- Техническая документация

**3. Резюмирование контента**
- Генерация отчётов
- Синтез документов
- Резюме заметок с совещаний

**4. Запросы к предметной области**
- Поиск медицинской информации
- Поиск юридических документов
- Поиск научных статей
- Поиск фрагментов кода

### ❌ Когда нужно больше

**Сложные запросы, требующие нескольких проходов извлечения:**
- «Сравните функцию X в продуктах A, B и C»
- Требуется: извлечение с несколькими запросами

**Низкое начальное качество извлечения:**
- Неудачная модель эмбеддингов для предметной области
- Требуется: переупорядочивание, расширение запросов

**Необходимость высокой точности (precision):**
- Критические решения (медицина, юриспруденция)
- Требуется: переупорядочивание, фильтрация, верификация

**Крупные базы знаний:**
- Миллионы документов
- Требуется: фильтрация, предварительная фильтрация на основе метаданных

## Измерение успеха

### Метрики качества

**1. Точность извлечения (precision)**
```
Precision = Relevant Retrieved / Total Retrieved
```
Цель: > 80% для k=3

**2. Полнота извлечения (recall)**
```
Recall = Relevant Retrieved / Total Relevant in DB
```
Цель: > 60% для k=5

**3. Точность ответа**
```
Accuracy = Correct Answers / Total Questions
```
Цель: > 90% для запросов в пределах предметной области

**4. Удовлетворённость пользователей**
- Оценки «нравится / не нравится» на ответы
- Частота уточняющих вопросов
- Частота отказа от запроса

**1. Сквозная задержка**
- Цель: < 1 секунда для хорошего пользовательского опыта

**2. Разбивка по компонентам**
- Время эмбеддингов

**3. Пропускная способность**
- Запросов в секунду
- Количество поддерживаемых одновременных пользователей

## Резюме

### Что такое базовое извлечение?

**Основная идея:** Извлечение релевантных документов с помощью векторного поиска, а затем генерация ответов с использованием LLM и контекста.

**Три шага:**
1. Извлечение: поиск по векторному сходству для top-k документов
2. Формирование: документы форматируются в строку контекста
3. Генерация: LLM создаёт ответ, основанный на контексте

### Ключевые параметры

- **k**: количество документов для извлечения (по умолчанию: 3)
- **Порог сходства**: минимальная оценка для включения (по умолчанию: 0,3–0,5)
- **Максимальный размер контекста**: лимит токенов для LLM (зависит от модели)

### Почему это работает

✅ Основывает LLM на фактах (снижает галлюцинации)  
✅ Добавляет специфические знания предметной области  
✅ Поддерживает обновления в реальном времени  
✅ Обеспечает указание источников  

### Когда этого достаточно

✅ Внутренние базы знаний  
✅ Системы Q&A  
✅ Поиск по документам  
✅ Запросы к предметной области  

### Когда нужны продвинутые стратегии извлечения

❌ Сложные многоэтапные запросы  
❌ Низкое начальное качество извлечения  
❌ Высокие требования к точности (precision)  
❌ Очень крупные базы знаний  

### Лучшие практики

1. Начните с k=3
2. Установите порог сходства (0,4)
3. Контролируйте качество извлечения
4. Грамотно обрабатывайте граничные случаи
5. Тестируйте с RAG и без него
6. Логируйте и измеряйте всё
