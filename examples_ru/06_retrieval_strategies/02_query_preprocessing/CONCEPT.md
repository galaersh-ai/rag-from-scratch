# Предобработка запросов

## Что такое предобработка запросов?

Предобработка запросов — это **очистка пользовательских запросов** перед поиском релевантных документов.

Представьте это как подготовку ингредиентов перед приготовлением блюда:
- Помыть овощи (убрать шум)
- Нарезать всё одинакового размера (нормализовать формат)
- Использовать свежие ингредиенты (расширить сокращения)

**Простое определение:**
> Предобработка запросов преобразует неструктурированный пользовательский ввод в чистые, единообразные запросы, которые работают лучше при поиске.

---

## Зачем это нужно?

### Проблема

Пользователи вводят запросы по-разному:

```
"What's ML?"
"WHAT IS ML!!!"
"what   is    ml"
"  What is ML?  "
```

**Все эти запросы означают одно и то же**, но компьютер воспринимает их как разные!

### Без предобработки

```
Пользователь 1: "What's ML?"     → Получает результат A
Пользователь 2: "what is ml"     → Получает результат B
Пользователь 3: "WHAT IS ML???"  → Получает результат C

❌ Один и тот же вопрос, разные ответы = Плохой пользовательский опыт
```

### С предобработкой

```
Пользователь 1: "What's ML?"     → "what is machine learning" → Результат X
Пользователь 2: "what is ml"     → "what is machine learning" → Результат X
Пользователь 3: "WHAT IS ML???"  → "what is machine learning" → Результат X

✅ Один и тот же вопрос, один и тот же ответ = Хороший пользовательский опыт
```

## Что такое «шум» в запросах?

**Шум** — это всё, что не помогает найти правильный ответ.

### Примеры шума

**1. Лишние пробелы**
```
"what   is    python"  ← Слишком много пробелов
"what is python"       ← Чисто
```

**2. КРИК (CAPS LOCK)**
```
"WHAT IS PYTHON!!!"    ← Заглавные буквы + знаки препинания
"what is python"       ← Чисто
```

**3. Избыток знаков препинания**
```
"what's python???"     ← Слишком много знаков препинания
"what is python"       ← Чисто
```

**4. Нечёткий формат**
```
"  What   IS   Python???  "  ← Пробелы, заглавные, знаки препинания
"what is python"             ← Чисто
```

### Почему шум вреден

```
Запрос с шумом:    "   WHAT   IS   Python???  "
                    ↓
              Модель эмбеддингов
                    ↓
         [0.234, 0.112, 0.889, ...]  ← Непоследовательный эмбеддинг
                    ↓
              Плохие результаты ❌


Чистый запрос:     "what is python"
                    ↓
              Модель эмбеддингов
                    ↓
         [0.891, 0.234, 0.567, ...]  ← Последовательный эмбеддинг
                    ↓
              Хорошие результаты ✅
```

## Техники предобработки

### 1. Приведение к нижнему регистру (использовать всегда)

**Что делает:** Преобразует весь текст в нижний регистр

```javascript
"PYTHON" → "python"
"Python" → "python"
"PyThOn" → "python"
```

**Почему это помогает:**

Все эти варианты должны означать одно и то же:
- "Python programming"
- "PYTHON programming"
- "python programming"

Но компьютер видит их как разные. Приведение к нижнему регистру делает их одинаковыми!

**Пример:**
```javascript
// До
"What is Python?"
"WHAT IS PYTHON?"

// После приведения к нижнему регистру
"what is python?"  ← Теперь оба одинаковые!
"what is python?"
```

**Правило:** Всегда используйте нижний регистр для пользовательских запросов.

---

### 2. Нормализация пробелов (использовать всегда)

**Что делает:** Исправляет проблемы с пробелами

```javascript
"what   is    python"     → "what is python"
"what\tis\npython"        → "what is python"
"  what is python  "      → "what is python"
```

**Почему это помогает:**

Лишние пробелы сбивают систему с толку:
- Компьютер видит: "what␣␣␣is␣␣python"
- Должно быть: "what␣is␣python"

**Как это работает:**
1. Заменить все пробельные символы (пробелы, табуляции, переносы строк) на один пробел
2. Убрать пробелы в начале и в конце

**Пример:**
```javascript
// До
"   What   IS   Python???  "

// После нормализации
"what is python???"  ← Чистые пробелы
```

---

### 3. Расширение запроса (рекомендуется)

**Что делает:** Заменяет сокращения на полные слова

```javascript
"ML"  → "machine learning"
"AI"  → "artificial intelligence"
"JS"  → "javascript"
"plz" → "please"
```

**Почему это помогает:**

Пользователи используют сокращения, а документы — полные термины:

```
Пользователь вводит:  "Tell me about ML"
Документ содержит:     "Machine learning is a subset of AI..."

Проблема: "ML" не очень хорошо соответствует "machine learning"!

Решение: Расширить "ML" → "machine learning"
```

**Типичные расширения:**

**Технические:**
```javascript
ml   → machine learning
ai   → artificial intelligence
api  → application programming interface
js   → javascript
py   → python
```

**Разговорные:**
```javascript
plz  → please
thx  → thanks
abt  → about
diff → difference
```

**Пример:**
```javascript
// До
"tell me abt ML plz"

// После расширения
"tell me about machine learning please"

// Результат: Гораздо лучшее соответствие документам!
```

---

### 4. Удаление специальных символов (использовать осторожно)

**Что делает:** Удаляет знаки препинания и символы

```javascript
"What's Python???" → "what s python"
"###ML###"        → "ml"
"python!!!"       → "python"
```

**Почему это помогает:**

Специальные символы добавляют шум:
- Они мало меняют смысл
- Делают эмбеддинги непоследовательными
- Разные знаки препинания = разные эмбеддинги

**Когда это помогает:**
```javascript
// Избыточный шум — удаление помогает!
"###python### !!! @@@" → "python"
✅ Гораздо чище
```

**Когда это вредит:**
```javascript
// Естественная пунктуация — удаление вредит!
"What's machine learning?" → "what s machine learning"
❌ Создан артефакт "s"
```

**Решение:**

Расширение В ПЕРВУЮ ОЧЕРЕДЬ, затем удаление специальных символов:

```javascript
// ПРАВИЛЬНЫЙ ПОРЯДОК ✅
"What's ML"
  → расширение: "what is machine learning"
  → удаление специальных символов: "what is machine learning"
  ← Чисто!

// НЕПРАВИЛЬНЫЙ ПОРЯДОК ❌
"What's ML"
  → удаление специальных символов: "what s ml"
  → расширение: "what s machine learning"
  ← Артефакт "s" остаётся!
```

## Рекомендуемый конвейер

### Пошаговый процесс

Вот оптимальный порядок очистки запросов:

```javascript
function preprocessQuery(query) {
    // 1. Привести к нижнему регистру
    query = query.toLowerCase();

    // 2. Убрать пробелы по краям
    query = query.trim();

    // 3. Исправить пробелы
    query = query.replace(/\s+/g, ' ');

    // 4. Расширить сокращения (ДО удаления специальных символов!)
    query = expandAbbreviations(query);

    // 5. Удалить специальные символы
    query = query.replace(/[^a-z0-9\s]/gi, ' ');

    // 6. Ещё раз исправить пробелы
    query = query.replace(/\s+/g, ' ').trim();

    return query;
}
```

### Реальный пример

Давайте посмотрим, как это работает:

```javascript
Вход: "   Hey!!! Tell me abt ML plz???  "

Шаг 1 — Привести к нижнему регистру:
"   hey!!! tell me abt ml plz???  "

Шаг 2 — Убрать пробелы по краям:
"hey!!! tell me abt ml plz???"

Шаг 3 — Исправить пробелы:
"hey!!! tell me abt ml plz???"

Шаг 4 — Расширить сокращения:
"hey!!! tell me about machine learning please???"

Шаг 5 — Удалить специальные символы:
"hey tell me about machine learning please"

Шаг 6 — Ещё раз исправить пробелы:
"hey tell me about machine learning please"

✅ Итог: Чистый, пригодный для поиска запрос!
```

### Почему именно такой порядок?

**Порядок важен!** Представьте это как приготовление блюда:

❌ **Неправильно:** Нарезать овощи, а потом помыть их
- Грязь попадёт внутрь нарезки!

✅ **Правильно:** Помыть овощи, а потом нарезать их
- Чистые ингредиенты с самого начала

**В предобработке:**

❌ **Неправильно:** Удалить специальные символы, а потом расширить
```
"What's ML" → "what s ml" → "what s machine learning"
                            ↑ Артефакт остаётся!
```

✅ **Правильно:** Расширить, а потом удалить специальные символы
```
"What's ML" → "what is machine learning" → "what is machine learning"
                                           ← Чисто!
```

### Проблема 3: Несовпадение словарного запаса

**Разрыв:**
```
Запрос пользователя:     Содержимое документа:
"Tell me about ML"       "Machine learning is a subset of AI..."
"JS frameworks"          "JavaScript libraries for building UIs..."
"abt APIs plz"           "About application programming interfaces..."
```

**Влияние:**
- Словарный запас пользователя ≠ Словарный запас документа
- Сокращения не соответствуют полным терминам
- Разговорная речь не соответствует формальному стилю
- Семантическое сходство маскируется лексическими различиями

### Проблема 4: Дискуссия о стоп-словах

**Традиционный подход (TF-IDF, BM25):**
```
"what is the best way to learn python"
       ↓ Удалить стоп-слова
"best way learn python"
✓ Фокус на значимых словах
✓ Снижение шума
```

**Современный подход (нейронные эмбеддинги):**
```
"what is the best way to learn python"
       ↓ Сохранить стоп-слова
"what is the best way to learn python"
✓ Контекст важен: "what is" vs "what"
✓ Грамматика даёт семантические подсказки
✓ Модели обучены на естественном языке
```

**Дилемма:**
- Старые методы рекомендуют удалять
- Новые модели рекомендуют сохранять
- Влияние зависит от конкретной модели
- Необходимо тестирование на практике

## Ключевые концепции

### Концепция 1: Стабильность пространства эмбеддингов

**Определение:**
Степень, в которой семантически идентичные запросы порождают похожие эмбеддинги.

**Визуализация:**
```
Нестабильное пространство эмбеддингов (без предобработки):
┌────────────────────────────────────┐
│  "python"                          │
│         ●                          │
│                                    │
│                    "Python"        │
│                           ●        │
│                                    │
│  "PYTHON!!!"                       │
│         ●                          │
└────────────────────────────────────┘
Эмбердинги разбросаны, несмотря на одинаковый смысл

Стабильное пространство эмбеддингов (с предобработкой):
┌────────────────────────────────────┐
│                                    │
│                                    │
│          Кластер "python"          │
│                 ●●●                │
│                                    │
│                                    │
└────────────────────────────────────┘
Эмбеддинги плотно сгруппированы
```

**Почему это важно:**
- Предсказуемость извлечения
- Единообразный пользовательский опыт
- Меньше пограничных случаев
- Проще отлаживать

**Как измерить:**
```javascript
// Вычислить косинусную сходство между вариациями запроса
const queries = ["What is Python?", "WHAT IS PYTHON", "what is python"];
const embeddings = queries.map(q => embed(preprocess(q)));

// Высокая стабильность: сходство > 0.99
// Низкая стабильность: сходство < 0.95
const stability = cosineSimilarity(embeddings[0], embeddings[1]);
```

### Концепция 2: Соотношение сигнала к шуму

**Определение:**
Соотношение осмысленной семантической информации к нерелевантному поверхностному шуму в запросе.

**Формула:**
```
SNR = Семантическая информация / Общая информация

Высокий SNR:   "machine learning algorithms"  (всё содержательное)
Средний SNR:   "what is ML?"                  (немного шума, немного содержательного)
Низкий SNR:    "###ML!!!"                     (в основном шум)
```

**Влияние на эмбеддинги:**
```
Запрос с высоким SNR:
[0.23, 0.87, 0.45, ...]  ← Чёткий семантический сигнал
         ↓
    Хорошее извлечение

Запрос с низким SNR:
[0.12, 0.09, 0.11, ...]  ← Слабый, зашумлённый сигнал
         ↓
    Плохое извлечение
```

**Цель предобработки:**
Повысить SNR, удаляя шум и сохраняя полезный сигнал.

### Концепция 3: Сохранение семантики

**Определение:**
Степень, в которой предобработка сохраняет исходное семантическое намерение запроса.

**Баланс:**
```
Слишком мало предобработки:
"###What's ML???###" → Сохранить всё → Высокий шум, низкое качество

Оптимальная предобработка:
"###What's ML???###" → "what is machine learning" → Низкий шум, высокое качество

Слишком много предобработки:
"###What's ML???###" → "ml" → Потерян контекст, низкое качество
```

**Кривая компромисса:**
```
Качество извлечения
       ↑
   100%│     ╱╲
       │    ╱  ╲
       │   ╱    ╲
       │  ╱      ╲___
       │ ╱           ╲___
     0%└────────────────────→ Агрессивность предобработки
       Нет    Мягкая  Оптимальная  Агрессивная  Экстремальная
```

**Ключевой вывод:**
Больше предобработки — не всегда лучше. Найдите оптимальную точку для вашего конкретного случая.

### Концепция 4: Адаптация к предметной области

**Определение:**
Настройка предобработки под словарный запас, стиль и соглашения вашей конкретной предметной области.

**Универсальная предобработка:**
```javascript
expansions = {
    'ml': 'machine learning',
    'ai': 'artificial intelligence',
    'api': 'application programming interface'
}
// Подходит для общих технических запросов
```

**Почему это важно:**
- Универсальная предобработка не учитывает особенности предметной области
- Адаптация к предметной области повышает качество извлечения на 15–30%
- Пользователи ожидают поведения, соответствующего предметной области
---

**Заключение:**

Предобработка запросов — это и искусство, и наука. Искусство — понимать пользователей и предметную область. Наука — измерять, что работает. Овладейте обоими аспектами, и вы создадите системы RAG, которые впечатляют пользователей релевантными и единообразными результатами.
